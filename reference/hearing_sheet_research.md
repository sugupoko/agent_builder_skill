# ヒアリングシート: リサーチ系エージェント

> **対象**: 週次ニュース配信、競合動向ウォッチ、論文サーベイ、業界トレンド分析など、**外部情報を集めて要約・配信**するタイプのエージェント

最初に `hearing_templates.md` の共通項目を埋めてから、本シートで詳細化する。
このシートを印刷して関係者に記入してもらうのが最速。

---

## 1. テーマ定義

### 1-1. テーマ名（1行）
> 例: 「業界週次レポート」

### 1-2. テーマの守備範囲（3〜5行）
何を含み、何を含まないか。

```
含む:
- ○○分野の新製品・新サービス
- ○○社の動向
- 政策・規制動向

含まない:
- 株価・決算（決算は経営動向セクションで分離）
- エンタメ・スポーツ
```

### 1-3. このテーマで扱う「カテゴリ」を3〜8個列挙

カテゴリは出力レポートの章立てになる。

| カテゴリ名 | 内容 | 該当例 |
|---|---|---|
| 例: 製品・承認 | 新製品ローンチ、規制承認 | 「○○社が新サービスをローンチ」 |
| 例: 提携・投資 | M&A、業務提携、資金調達 | 「A社とB社が共同開発で提携」 |
| 例: 規制・標準化 | 政府・業界団体の動き | 「業界ガイドライン改訂」 |
| 例: 論文 | 学術文献 | 「○○誌に新手法掲載」 |
| 例: 経営動向 | 決算・業績・人事 | 「○○社が業績下方修正」 |
| 例: インシデント | 不祥事・障害・セキュリティ | 「○○社で情報漏えい」 |

## 2. 情報源（ソース）

### 2-1. 既知の情報源を列挙

国内・海外、無料・有料、定期発信されているか、を表で:

| 名称 | URL/API | 言語 | 更新頻度 | アクセス | カテゴリ |
|---|---|---|---|---|---|
| 例: Google News | RSS | ja/en | リアルタイム | 無料 | 業界全般 |
| 例: PR TIMES | RSS | ja | 随時 | 無料 | 製品リリース |
| 例: 業界ニュース誌 | スクレイピング | ja | 日次 | 有料 | 業界 |
| 例: PubMed | E-utilities | en | 日次 | 無料 | 論文 |
| 例: 政府サイト | RSS | ja | 不定期 | 無料 | 規制 |

### 2-2. 信頼できないソース（blocklist候補）
誤マッチや低品質で除外したいソース・出版社を列挙。

```
- 個人ブログ全般
- 株価自動生成サイト（TipRanks/MarketBeat 等）
- ゲーム・エンタメサイト
- アフィリエイト中心のまとめサイト
```

### 2-3. キーワード設計

カテゴリ別に、**含めるキーワード**と**誤マッチ防止の除外語**を整理:

```yaml
カテゴリ: 製品・承認
  含めるキーワード:
    - "新製品"
    - "ローンチ"
    - "承認"
  除外キーワード:
    - "○○ゲーム"  # 同名の製品があり混同しがち
```

### 2-4. 主要プレイヤー（watchlist）

固定で監視したい企業・団体・人物を 5〜10 個。

| エンティティ | ヒット用キーワード（誤マッチ回避を意識） |
|---|---|
| 例: A社 | "A株式会社", "A Inc.", "A Holdings" |
| 例: B団体 | "B協会", "B Foundation" |

## 3. 編集者の声（最重要）

> ここに業務担当者の判断スキルを言語化して書き込む。LLM のプロンプトに直接注入される。

### 3-1. ペルソナ（あなた自身を3〜5行で）
あなたの経験年数、専門領域、立ち位置を書く。

```
例: 15年の○○業界経験を持つ編集者。
A領域とB領域の両方の視点を持つ。
規制動向と現場運用の橋渡しが得意。
```

### 3-2. 視点・スタンス（3〜5項目）
何を見るか、何を問うか。

```
- 「で、ウチはどう動くか」を常に問う実務志向
- 広報的成功談より現場のつまずきに着目
- 海外トレンドは「いつ国内に来るか」の視点で評価
- 単発の出来事より構造的変化を読み取る
```

### 3-3. 執筆ルール（3〜5項目）
書いてほしくないこと、必ず含めてほしいこと。

```
- 「業界が活況」「期待が高まる」のような汎用表現は禁止
- 数字・固有名詞・日付を必ず含める
- 同案件の重複ソースは1件にマージ
- 読者が今週・来週・来月にすべきアクションを最低3件挙げる
```

## 4. 出力形式

### 4-1. メール件名のフォーマット
```
例: [○○週報] {ヘッドライン} ({日付})
```
ヘッドラインは LLM が動的生成する。

### 4-2. 本文の構成（必要なセクションを ✓）

| セクション | 必要 | 備考 |
|---|---|---|
| 目次 (## レベル) | □ | 自動生成 |
| 5分でわかる今週のポイント | □ | 4〜6項目、引用[N]付き |
| 編集者の見立て | □ | 200〜400字の独自視点 |
| 実務TODO（今週・来週・来月） | □ | 期限・引用付き |
| 出典リスト | □ | [N] 形式 |
| 主要プレイヤー動向ウォッチ | □ | watchlist 連動 |
| 数字で見る今週 | □ | 件数集計 |
| 来週以降の注目カレンダー | □ | タイトルから日付抽出 |
| 注目トピック背景調査 | □ | ReAct 深掘り |
| 補強リサーチ | □ | LLM が手薄な観点を判断 |

### 4-3. 配信頻度・配信先
- 頻度: 週1 / 隔週 / 月次 / 不定期
- 配信先: メール / Slack / Confluence / ファイル
- 配信日時: 例: 毎週月曜 8:00

## 5. 評価データ

### 5-1. 過去の良い出力の例（3件以上）
過去に書いた良いレポートのスクショ・URL・PDFを集める。LLM の評価で使う。

### 5-2. 悪い出力の例（3件以上）
「これはダメ」というパターン。汎用的すぎる、見出しが弱い、出典が無い、など。

### 5-3. 評価指標（3〜5個）
| 指標 | 計測方法 |
|---|---|
| 例: 件数 | 5分サマリの項目数（4〜6が目標） |
| 例: 数字含有率 | 各項目に数字が含まれている割合 |
| 例: 引用数 | 出典 [N] が3件以上付いている割合 |
| 例: コスト | 1実行あたり USD（上限設定） |
| 例: 所要時間 | 起動から配信完了まで（分） |

## 6. 運用条件

### 6-1. コスト
- 1回の実行予算: $0.05 / $0.30 / $1.00 / $...
- 月間予算: $5 / $30 / $100 / $...
- 軽量モード必要: はい / いいえ

### 6-2. 配信前レビュー
- 人間レビュー: 要 / 不要
- レビュアー: ○○さん
- 承認方法: Slackボタン / メール返信 / 手動再生成

### 6-3. キーワード・テーマ追加の主体
- YAMLを自分で編集できる: はい / いいえ
  - はい → そのまま運用
  - いいえ → 業務担当向け簡易UIが必要かも

---

## 7. 既知の落とし穴（リサーチ系特有）

事前に共有しておくと運用が滑らか:

- **重複の山**: 同じ案件が複数ソースから流れる → 正規化タイトルでマージ
- **時事のスパイク**: 大事件があると関連ニュースで埋まる → 件数バランス監視
- **キーワード誤マッチ**: 略称（"○○"が別業界用語と被る）→ exclude_keywords + 文脈ヒント
- **長期追跡記事**: 数週間続く案件で毎週同じ話題になる → 跨実行重複排除（SQLite）
