# ヒアリングシート: レビュー系エージェント

> **対象**: 申請書チェック、コード品質レビュー、契約書レビュー、文書のルール違反検出など、**ルール／基準に照らして対象を評価する**タイプのエージェント

`hearing_templates.md` の共通項目を埋めてから、本シートで詳細化する。

---

## 1. レビュー対象

### 1-1. 何をレビューするか
- 例: 経費申請書 / 契約書 / コードPR / 報告書 / 提案書 / 顧客メール

### 1-2. 入力形式
| 項目 | 値 |
|---|---|
| ファイル種別 | PDF / Word / Excel / Markdown / プレーンテキスト / JSON / API レスポンス |
| 1件あたりサイズ | 文字数 / ページ数 |
| 件数（日次/週次） | |
| 取得経路 | アップロード / API / メール / 共有フォルダ |

### 1-3. レビューの観点
複数ある場合は箇条書き:
- 例: ルールAに準拠しているか
- 例: 必須項目がすべて埋まっているか
- 例: 過去の類似案件と矛盾していないか
- 例: コンプライアンス上の問題がないか

---

## 2. ルール定義

### 2-1. ルールはどこにある？
- ドキュメント（PDF/Confluence）の構造化されたルール集
- 業務担当の頭の中（暗黙知）
- 過去事例の蓄積

### 2-2. ルールの粒度

| ルール種別 | 例 | 構造化のしやすさ |
|---|---|---|
| 必須項目チェック | 「日付欄が空でない」 | 易（コードで書ける） |
| 数値の範囲チェック | 「金額が10万円超なら承認必要」 | 易 |
| 文言一致チェック | 「禁止語リストにヒット」 | 易 |
| 文脈判断 | 「この提案理由は妥当か」 | 難（LLM 必須） |
| 過去事例との整合性 | 「先月の類似案件と矛盾しないか」 | 難（RAG + LLM） |

→ **コードで書けるルールは最初にコードで実装する**。LLM は文脈判断・過去事例との整合性に集中。

### 2-3. ルールの抽出（可能なら自動化）
- 既存ルールドキュメントから RAG で参照
- ルールを YAML や JSON に切り出して LLM プロンプトに含める
- 業務担当者が「最近のレビューで気づいた追加ルール」をいつでも追記できる仕組み

## 3. レビュー結果の形式

### 3-1. 出力フォーマット
| 形式 | 用途 |
|---|---|
| **判定（合格/不合格/要相談）** | 単純な分岐 |
| **指摘リスト** | 改善点の箇条書き |
| **スコア + 理由** | 数値評価 + 根拠 |
| **赤入れ Markdown** | 元文書のどこが問題かを示す |
| **Slack/メール通知** | 即時共有 |

### 3-2. 例

```markdown
# レビュー結果: 申請書 #1234

## 判定: 要相談

## 指摘事項

1. **必須項目欠落** [重要度: 高]
   - 「決裁者承認」欄が空欄
   - ルール 3.2.1 違反

2. **金額の根拠不足** [重要度: 中]
   - 「概算」と記載されているが詳細見積もりが必要
   - ルール 4.1.5 で詳細見積もり必須

3. **過去事例との不整合** [重要度: 低]
   - 類似案件（#1100）では事前承認が取られていた
```

### 3-3. 配信先
- 申請者本人にメール
- 上長にCC
- 全件を Slack #review-log に
- DB に保存（監査用）

## 4. 編集者の声（レビュアーペルソナ）

> ここがレビュー系エージェントの肝。経験豊富なレビュアーの視点を言語化する。

### 4-1. ペルソナ
```
例: 10年経験の経理担当。
内部統制と業務効率の両立を重視。
過去の不正事例を多数見てきており、形式不備の裏に潜む実質的問題を見抜くのが得意。
```

### 4-2. 視点
```
- 形式不備があったら、その背景を疑う
- 金額より頻度パターンを重視する（小額の繰り返しが大金額より問題のことも）
- 申請者の負担軽減と内部統制のバランスを取る
```

### 4-3. ルール
```
- 全ての指摘に「ルール番号」を付ける（追跡可能性）
- 重要度（高/中/低）を必ず明示
- 改善方法を提案する（指摘だけで終わらない）
- 過剰なレビューで申請者を疲弊させない
```

## 5. 評価データ

### 5-1. 良いレビューの例（3件以上）
過去の良いレビュー結果を集める。

### 5-2. 悪いレビューの例（3件以上）
- 過剰指摘（細かすぎる）
- 漏れあり（重要な問題を見落とした）
- 根拠不明（ルール番号なし）

### 5-3. 評価指標
| 指標 | 計測方法 | 目標 |
|---|---|---|
| 重要指摘の網羅率 | 人間レビューと比較した一致率 | 90%+ |
| 過剰指摘率 | 指摘のうち不要だったもの | 10%以下 |
| ルール番号付与率 | 指摘にルール番号がある率 | 100% |
| 平均レビュー時間 | 1件あたり所要秒数 | 30秒以下 |
| コスト | 1件あたり USD | $0.05以下 |

## 6. 失敗時の対処

### 6-1. LLM が判断を間違えた場合
- 申請者からの異議申立フロー
- 業務担当者の上書き（手動オーバーライド）
- 失敗ケースを評価データに追加して改善

### 6-2. 重大なエラーの想定
- 個人情報を誤って外部送信 → 個人情報フィールドを LLM に送らない
- 不正案件を見逃した → 抜き取り検査を運用
- 大量の誤検知で業務停止 → 信頼度しきい値の設定

---

## 7. 設計パターン選定の参考

レビュー系で典型的な構成:

```
intake (受領) → classify (分類) → extract_rules (該当ルール抽出, RAG) → judge (LLM判定) → format (整形) → send
                  ↑                                                    ↑
                  Haiku                                             Sonnet（精度重視で Reflection 推奨）
```

- **Reflection** は重要: 「最初の判定をもう一度見直す」が誤検知を減らす
- **Tool Use**: 過去事例DBから類似案件を引いて参考にする
- **Multi-Agent** は基本不要（単一観点でレビューできる場合）
