# 編集者の声をプロンプトに注入する

LLM 出力が「業界が活況」「期待が高まる」のような汎用表現になる症状の特効薬。
業務担当者の判断スキルを言語化して **editorial.persona / perspective / rules** の3要素にし、システムプロンプトに注入する。

---

## なぜこれが効くのか

LLM はデフォルトで「広く浅く」書く傾向がある。
「あなたは20年経験の編集者です」と人格を与えるだけで、文体・着眼点が劇的に変わる。

LayerX の AI オンボーディング記事でも、「LLMに求める役割やゴールを明確化し、専門知識・情報資産・業務手順を提供する」ことの重要性が強調されている。

---

## 3要素の役割

### persona — 「誰として書くか」

3〜5行で人格を書く:
- 経験年数
- 専門領域
- 立ち位置・スタンス

```yaml
persona: |
  20年の○○業界経験を持つ編集者。
  △△ と □□ の両方の視点を持つ。
  現場の運用課題と経営判断の橋渡しが得意。
```

### perspective — 「何を見るか」

3〜5項目で視点を列挙:

```yaml
perspective: |
  - 「で、ウチはどう動くか」を常に問う実務志向
  - 広報的成功談より、現場のつまずきに着目
  - 海外トレンドは「いつ国内に来るか」の視点で評価
  - 単発の出来事より構造的変化を読み取る
```

### rules — 「何を守るか」

3〜5項目で執筆ルールを列挙（特に「禁止事項」が効く）:

```yaml
rules: |
  - 「業界が活況」「期待が高まる」のような汎用表現は禁止
  - 数字・固有名詞・日付を必ず含める
  - 同案件の重複ソースは1件にマージ
  - 「読者が今週・来週・来月にすべきアクション」を最低3件挙げる
```

---

## プロンプトへの注入方法

```python
def build_prompt(state):
    cfg = state["cfg"]
    editorial = cfg.get("editorial", {}) or {}
    persona = editorial.get("persona", "")
    perspective = editorial.get("perspective", "")
    rules = editorial.get("rules", "")
    
    persona_block = f"あなたのペルソナ:\n{persona}\n\n" if persona else ""
    perspective_block = f"視点・スタンス:\n{perspective}\n\n" if perspective else ""
    rules_block = f"執筆ルール:\n{rules}\n\n" if rules else ""
    
    prompt = f"""次のデータを編集してください。

{persona_block}{perspective_block}{rules_block}データ:
{data}

出力フォーマット:
..."""
    return prompt
```

## 良いペルソナの引き出し方（ヒアリング）

業務担当者に質問する具体例:

### Q1. 経験を引き出す
- 「この業界に何年いますか？」
- 「これまでに見てきた最大の変化は？」
- 「他社の編集者と違う、あなたの強みは？」

### Q2. 視点を引き出す
- 「経験豊富な担当者が見て『これは外せない』と思う記事の特徴は？」
- 「『広報的すぎる発表』と『本物の動き』をどう見分けてる？」
- 「で、ウチはどう動くか、を考えるとき、どの観点を優先する？」

### Q3. ルールを引き出す
- 「絶対に書きたくない表現は？」
- 「読者が読み終わったときに何を持ち帰ってほしい？」
- 「『これを書かなかったら失敗』というレベルの必須要素は？」

## ペルソナの効果検証

ペルソナ注入の有無で出力を比較:

### Before（ペルソナなし）
```
- 業界では生成AIへの注目が高まっている
- 各社が新サービスをローンチしている
- 競争が激化している
```

### After（ペルソナあり）
```
- A社 Q1売上+36%、新サービス導入数+500%。プラットフォーム戦略が加速 [3, 7]
- B社が公的保険網へ侵食開始、低中所得国への展開テンプレ化が進む [10]
- 業界編成は「単品検査」から「データプラットフォーム」へ移行中 [5, 12, 18]
```

数字・固有名詞・構造的観察が具体化される。

## 編集者の見立て（Editor Take）セクション

サマリだけでなく**「編集者の主観的解釈」を独立セクションとして書かせる**:

```python
prompt = f"""...

出力構造:
## SUMMARY
（5分でわかる今週のポイント）

## EDITOR_TAKE
（編集者の見立て。1〜2段落で200〜400字。
「今週の動きから読み取れる構造的変化」「半年〜1年先に何が起こりうるか」
を編集者の主観として書く。汎用的・空虚な表現は禁止）
"""
```

これでアナリストが書くようなコメントが出るようになる。

## 落とし穴

### ペルソナを抽象的にすると効果が薄い

NG: 「経験豊富な編集者として書いてください」
OK: 「20年の○○業界経験を持つ編集者。△△ に詳しく、現場の□□に共感する立場」

### perspective を多すぎる項目にしない

10項目もあると LLM が散漫になる。**3〜5項目**に絞る。

### rules で「禁止事項」を明示する

「書いてほしいこと」より「書いてほしくないこと」の方が効く:
- ❌「具体的に書いてください」
- ✅「『業界が活況』のような汎用表現は禁止」

### ペルソナを定期的に見直す

ヒアリング後の最初の運用で出力を見て、追加すべき視点・ルールを発見する。
最初から完璧なペルソナは作れない。**運用しながらブラッシュアップ**する前提で。

---

## ⚠️ プロンプト肥大化の罠 (実体験ベースの警告)

editorial.persona / perspective / rules を磨く段階で **「追加」よりも「整理・削減」を優先**する。

### 実プロジェクトで観測した失敗 (cs_triage_agent v2)

v1 の LLM Judge スコアで弱点 (persona_fit 2.96, info_completeness 2.91) を見つけたので「persona を立体化」「rules に新概念を追加」「6セクションに構造化」したところ、**逆に総合スコア 3.52 → 3.34、数値正確性 4.04 → 3.70 と大幅劣化**した。

→ プロンプトが長くなりすぎると LLM が要点を見失う。Anthropic 公式 "Be specific, not extensive" の通り。

### 改善アプローチの良し悪し

| ❌ やりがちな悪手 | ✅ 効く改善 |
|---|---|
| persona の人物像を細かく描写 (経歴・性格・信条) | persona は 3-5 行で人格の核だけ |
| rules を 6 セクションに構造化 | フラットな箇条書きで 7-10 項目以内 |
| 「最重要」「絶対」を多用 | 1-2 個だけ本当に最重要にする |
| 新ルールを追加 | 既存ルールの優先順位を明確化 |
| 「rules を増やす」 | **Few-shot 例 2-3 件で代替**を検討 |
| 「合格件数」で成否判断 | **LLM Judge で次元別スコア比較**で判断 |

### 改善前の必須チェック
1. 現在のプロンプトで LLM Judge ベースラインを取る (5次元スコア)
2. 改善案を実装する
3. 同じ評価データで再 judge し、次元別の増減を確認
4. 1つでも次元スコアが下がっていたら一旦戻す or 別案を検討

詳細事例: `workspace/cs_triage_agent/v2/reports/v1_vs_v2_comparison.md`

## まとめ

- **3要素**: persona（誰）/ perspective（何を見るか）/ rules（何を守るか）
- **YAMLに書く**: 業務担当が編集できる粒度に
- **ヒアリングで引き出す**: 「どう見分ける？」「で、どう動く？」
- **禁止事項を明示**: 「○○のような表現は禁止」
- **運用しながら磨く**: 最初から完璧は作れない
- **磨き方は「追加」より「整理・削減」**: LLM Judge で必ず検証
