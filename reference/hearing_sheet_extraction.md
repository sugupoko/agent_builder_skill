# ヒアリングシート: 抽出系エージェント

> **対象**: メール添付ファイルの取得、文書からの構造化情報抽出、エンティティ抽出、表データの正規化など、**ソースから特定情報を取り出して構造化する**タイプのエージェント

`hearing_templates.md` の共通項目を埋めてから、本シートで詳細化する。

---

## 1. 抽出対象

### 1-1. 何から、何を抽出するか
| 入力ソース | 抽出するもの |
|---|---|
| 例: 顧客からのメール本文 | 注文番号 / 数量 / 配送希望日 |
| 例: PDF 領収書 | 日付 / 金額 / 取引先 / 品目 |
| 例: ニュース記事 | 企業名 / 製品名 / 発表日 / 業績数値 |
| 例: 申込書 | 氏名 / 住所 / 連絡先 / 希望サービス |

### 1-2. 入力フォーマット
- ファイル種別: PDF / 画像（OCR必要）/ HTML / メール / API レスポンス
- 構造化度: 完全フリーフォーム / 半構造化（決まったフォーマット）/ 構造化（テーブル）
- 例外パターン: 想定外フォーマットの頻度・対処

### 1-3. 出力フォーマット
```yaml
# JSON Schema または例
order_id: string  # 必須
quantity: integer  # 必須
ship_date: date  # 必須
customer_note: string  # 任意
```

---

## 2. 抽出フィールドの詳細

各フィールドについて:

| フィールド名 | 型 | 必須 | バリデーション | 例 |
|---|---|---|---|---|
| order_id | string | ✓ | 正規表現 `[A-Z]{3}-\d{6}` | "ABC-123456" |
| quantity | integer | ✓ | > 0, < 10000 | 50 |
| ship_date | date | ✓ | 今日以降 | "2026-05-20" |
| customer_note | string | × | 最大500文字 | "急ぎでお願いします" |

### バリデーション失敗時の挙動
- 必須フィールドが取れない → 失敗扱い、人間に回す
- 形式不正 → 警告つきで取り込み or 失敗
- 範囲外 → 警告つきで取り込み

---

## 3. 抽出戦略

### 3-1. ハイブリッド構成

```
受領 → 前処理（OCR等）→ 分類 → 抽出（LLM）→ バリデーション → 構造化保存
                              ↑                ↑
                            Tool Use      コードベース
                            （複雑な      （正規表現・型）
                             文書なら
                             RAG等）
```

### 3-2. LLM か正規表現か

| ケース | 推奨 |
|---|---|
| 完全に構造化された表 | コード（pandas / openpyxl） |
| 半構造化（決まったテンプレ） | コード + 補助的に LLM |
| 自由記述 | LLM 主導 |
| 画像 | OCR + LLM |

**鉄則**: 正規表現で書ける部分は正規表現で。LLM は最後の手段。

### 3-3. プロンプト設計

抽出を構造化出力で:

```python
prompt = f"""次の文書から以下のフィールドを JSON で抽出してください。

文書:
{document}

抽出フィールド:
- order_id (string, 必須): 注文番号。"ABC-123456" のような形式
- quantity (integer, 必須): 数量
- ship_date (date, 必須): 配送希望日（YYYY-MM-DD）
- customer_note (string, 任意): 顧客からのメモ

JSONのみ返す。前置き禁止。フィールドが見つからない場合は null。"""
```

### 3-4. ツール使い

複雑な抽出では:
- **lookup_db**: 抽出した値を既存DBで検証（顧客IDが存在するか等）
- **fetch_related**: 文中で参照されている関連文書を取得
- **calculator**: 数値の計算が必要な場合（小計＋消費税等）

## 4. エラー処理

### 4-1. 抽出失敗のパターン

| 失敗 | 対処 |
|---|---|
| OCR が崩れる | 元画像を人間に回す、もしくは別 OCR エンジンで再試行 |
| 必須フィールドが取れない | 失敗扱い、ユーザーに通知 |
| 形式違反（正規表現に合わない） | 警告つき取り込み、人間レビュー |
| LLM がでたらめな値を返す | バリデーションで弾く、再試行 |

### 4-2. 信頼度スコア

LLM に確信度も返させる:
```python
prompt += "\n各フィールドに confidence: 0-1 を付けてください。"

# 0.8未満は人間レビューに回す
if any(v["confidence"] < 0.8 for v in extracted.values()):
    queue_for_human_review(extracted)
```

## 5. 編集者の声（抽出ルール作者ペルソナ）

抽出系では「文書をどう読み解くか」のルールを言語化する:

```yaml
persona: |
  10年経験のオペレーター。
  顧客の様々な書き方を見てきており、文脈から正しい情報を読み取るのが得意。

perspective: |
  - 表記ゆれ（「2026/5/20」「2026年5月20日」「5/20」）を正規化する
  - 顧客が省略しがちな項目（年・敬称等）を文脈から補う
  - あいまいな記載は人間に回す（無理に推測しない）

rules: |
  - 確信度 0.8 未満なら人間レビューフラグを立てる
  - 過去パターンと一致しない記載は警告
  - 数値は必ずバリデーションを通す
```

## 6. 評価データ

### 6-1. 良い抽出の例（10件以上）

「正解ペア」を作る:
- 入力文書（オリジナル）
- 期待される構造化出力（JSON）

### 6-2. 評価指標

| 指標 | 計測 | 目標 |
|---|---|---|
| フィールド毎の精度 | 各フィールドが正解と一致した率 | 95%+ |
| 全体の完全一致率 | 全フィールドが正解と一致 | 80%+ |
| バリデーション pass 率 | 出力がスキーマに合致 | 99%+ |
| 人間介入率 | 信頼度低 で人間レビューに回した率 | 10%以下 |
| 1件あたりコスト | USD | < $0.10 |

### 6-3. 評価方法

```python
def eval_extraction(predicted, ground_truth):
    field_match = {
        k: predicted.get(k) == ground_truth.get(k)
        for k in ground_truth
    }
    full_match = all(field_match.values())
    return {
        "field_accuracy": sum(field_match.values()) / len(field_match),
        "full_match": full_match,
        "missing_fields": [k for k, v in field_match.items() if not v],
    }
```

## 7. 構造化保存先

|---|---|

| **JSON ファイル** | 軽量・バッチ処理 |

## 8. 設計パターン選定の参考

抽出系で典型的な構成:

```
intake → ocr (必要時) → classify (分類, Haiku) → extract (LLM, Sonnet) → validate (コード) → store
                                                       ↑
                                                  Tool Use（DB照合・関連文書取得）
```

- **Reflection は重要**: 抽出結果の self-check で精度向上
- **Tool Use** で既存 DB との照合
- **Multi-Agent は不要**（単純なケースが多い）
