# ヒアリング共通テンプレート

エージェント種別を問わず、最初に必ず確認する項目。
各種別の `hearing_sheet_*.md` の前段に位置するチェックリスト。

業務担当者・エンドユーザー・依頼者と対話しながら、これを **30〜60分** で埋める。

---

## A. 業務の理解

### A-1. その業務は何のため？
- このタスクが必要な理由（ビジネス目的）
- やめたら誰が困る？
- 今手作業で何時間かかっている？（週/月単位）

### A-2. 業務のステップを書き出す
担当者の作業を以下のように分解する。LLM 化したい範囲を明示。

```
1. [入力源] からデータを集める
2. [集めたもの] を [どう] 整理する
3. [整理結果] を [誰に] [どう] 渡す
4. [反応] を見て [次のアクション]
```

### A-3. どこに「人の判断スキル」があるか
- どの工程で経験・センスが効く？
- 新人ができないのは、どの判断？
- それを言語化できる？

→ **この回答が `editorial.persona` に入る**。重要項目。

---

## B. 入力と出力

### B-1. 入力
| 項目 | 質問 |
|---|---|
| 種類 | 自然言語 / 構造化データ / ファイル / API レスポンス / Slack メッセージ ... |
| 形式 | テキスト / JSON / CSV / PDF / 画像 ... |
| サイズ | 1回あたり何文字 / 何ファイル？ |
| 更新頻度 | リアルタイム / 1日1回 / 週次 ... |
| ソース | どこから取得？ API / RSS / DB / 手動アップロード |

### B-2. 出力
| 項目 | 質問 |
|---|---|
| 種類 | テキスト / Markdown / JSON / Excel / メール / Slack ... |
| 配信先 | メール / Slack / Confluence / ファイル保存 / DB ... |
| 形式 | 自由記述 / 定型レポート / 構造化（フィールド指定） |
| 量 | 1出力あたり何文字 / 何項目？ |
| 受け取る人 | 業務担当 / 管理職 / 顧客 / システム ... |

---

## C. 成功基準と評価

### C-1. 「うまくいった」の定義
- どうなれば成功？（時間削減？精度？網羅性？）
- 数値で表すと？（例: 週次レポート作成時間が3時間→30分）

### C-2. 評価データの確保
- **良い出力の例**を3つ以上書ける？ → スクショ/コピペで保存
- **悪い出力の例**を3つ以上書ける？ → 同上
- どこを見れば良し悪しが分かる？

### C-3. 評価方法（3軸のどれか）
- **人間評価**: 業務担当者がレビューする
- **LLM as a judge**: 別の LLM に良し悪しを判定させる
- **コードベース**: ルールベース（必須キーワード含むか / 文字数など）

---

## D. 制約と非機能要件

### D-1. コスト
- 1実行あたりの予算（USD / 円）
- 月間の上限
- 「軽量モード」を用意する必要あり？

### D-2. 速度
- 待てる時間: 即時 / 数秒 / 数分 / 数時間
- バッチで翌日でいいか？リアルタイム必須か？

### D-3. セキュリティ・コンプライアンス
- 扱うデータの機密度（公開/社内限/個人情報/規制対象）
- 外部 API（OpenAI/Anthropic/Google等）にデータを送って良いか？
- ログに何を残す（残さない）か？
- 監査要件はあるか？

### D-4. 依存関係
- 既に使っているツール / 既存システムとの連携
- 認証方式（SSO / OAuth / APIキー）
- 既存のメール/Slack/Confluenceの仕様

---

## E. 運用と継続性

### E-1. トリガー
- 手動 / cron / イベント駆動 / Webhook / その他
- 失敗時のリトライ・通知

### E-2. レビュー工程
- 配信前に人間レビュー要 / 不要
- レビュアーは誰？
- 承認のフロー（Slack ボタン？メール返信？）

### E-3. 変更管理
- キーワード調整・テーマ追加は誰がやる？（業務担当 / エンジニア）
- → **業務担当ができるなら YAML 駆動が必須**

### E-4. メンテナンス
- 誰が運用する？（個人 / チーム / 部署）
- ログ・コストの監視はどう？

---

## F. リスクと失敗モード

### F-1. 起こって困ること（重大度順）
- 個人情報漏えい
- 誤った情報を上司・顧客に送信
- コスト爆発（無限ループ）
- 配信が止まる

### F-2. 失敗時のフォールバック
- LLM が落ちたら？
- 外部 API が落ちたら？
- 出力が想定外の形式だったら？

---

## G. ユースケースカタログ（LayerX流）

業務担当者と一緒に、想定ユースケースを **30件程度**列挙する。
各ケースに **担当者の名前**を書くと、後で問い合わせやすい。

| # | ユースケース | 想定ユーザー | 想定頻度 | 価値 | 詳しい人 |
|---|---|---|---|---|---|
| 1 | 例: 週次の業界動向を5分で把握 | 経営企画 | 週1 | 競合監視の時間半減 | 田中 |
| 2 | 例: 申請内容のルール違反を即座にチェック | 経理 | 申請毎 | レビュー時間1/3 | 佐藤 |
| 3 | ... | ... | ... | ... | ... |

このリストは:
- どれから着手するか優先度判断に使う
- 後で別エージェントを派生させるときの素材になる
- ステークホルダー間で認識を揃える

---

## H. 完了判定

ヒアリング後、以下を全て答えられることを確認:

- [ ] 業務ステップが3〜10個に分解されている
- [ ] 入力と出力が具体的に書ける
- [ ] 「人の判断スキル」が言語化されている
- [ ] 評価データが10件以上ある（または収集予定がある）
- [ ] コスト・速度・セキュリティの制約が明確
- [ ] ユースケースが10件以上カタログ化されている
- [ ] リスクと対処が議論されている

これが揃えば `/agent-decompose` に進める。
