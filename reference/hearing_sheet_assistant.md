# ヒアリングシート: アシスタント系エージェント

> **対象**: 業務問い合わせ対応、社内 FAQ、ドキュメント検索アシスタント、お客様サポートなど、**ユーザーの質問に応答する**タイプのエージェント

`hearing_templates.md` の共通項目を埋めてから、本シートで詳細化する。

---

## 1. 利用シーン

### 1-1. 誰が、どんな状況で使う？

| 利用者 | 状況 | 期待 |
|---|---|---|
| 例: 新入社員 | 入社直後の業務理解 | 社内ルール・手順を即座に知りたい |
| 例: 管理職 | 部下からの質問対応 | 過去の方針・規定をすぐ確認 |
| 例: カスタマーサポート | 顧客対応中 | FAQ・既知問題を即座に検索 |
| 例: エンジニア | コーディング中 | 社内ライブラリ・実装パターンを参照 |

### 1-2. 質問のタイプ

- **事実検索**: 「○○の規定は何？」
- **手順確認**: 「○○の申請はどう進める？」
- **判断補助**: 「この場合、どう対処すべき？」
- **創作支援**: 「○○のメール文を書いて」

タイプによって設計が変わる:
- 事実検索 → RAG ベース、短い回答
- 判断補助 → 文脈を踏まえた推論、長めの回答
- 創作支援 → 自由生成、ペルソナが効く

## 2. ナレッジソース

### 2-1. 何を参照するか

| ソース | 形式 | アクセス |
|---|---|---|
| 例: 社内 Wiki | HTML | スクレイピング / API |
| 例: Confluence | API | OAuth / API トークン |
| 例: SharePoint | API | OAuth |
| 例: 過去の問い合わせログ | DB | SQL |
| 例: 社員 FAQ Excel | XLSX | 静的ロード |
| 例: 公開ドキュメント（製品マニュアル） | PDF | OCR + チャンク化 |

### 2-2. ナレッジの更新頻度

- 静的（年次更新）: 起動時に1回ロード
- 中頻度（週次）: 日次バッチで再インデックス
- リアルタイム: ベクトルDB に逐次追加

### 2-3. アクセス権限

| ナレッジ | 全員アクセス可 | 部署限定 | 個人限定 |
|---|---|---|---|
| 社内ルール | ✓ | | |
| 部署マニュアル | | ✓ | |
| 個人メモ | | | ✓ |

→ ユーザーの所属に応じてフィルタリング必要なら、認証連携が必須。

## 3. 応答スタイル

### 3-1. トーン
- 友好的 / フォーマル / 簡潔
- 絵文字使用 / 不使用
- 敬語レベル

### 3-2. 長さ

| 質問タイプ | 期待される回答長 |
|---|---|
| 事実検索 | 1〜3文 + 出典 |
| 手順確認 | ステップ箇条書き + 関連ドキュメントへのリンク |
| 判断補助 | 状況分析 + 選択肢の提示 |
| 創作支援 | 求められた成果物 |

### 3-3. 出典表示
- **必須**: 全ての応答に「どのドキュメントを参照したか」を明記
- 「分からない」と言える: ナレッジに無い質問には推測しない

## 4. 編集者の声（アシスタントペルソナ）

```yaml
persona: |
  あなたは ○○社の社内アシスタントです。
  業務知識に詳しく、規定・手順を正確に説明できます。
  分からないことは「分かりません」と言い、推測で答えません。

perspective: |
  - 質問者の状況を考慮して回答の粒度を調整する
  - 関連情報を1〜2個提示する（過剰提供は避ける）
  - 行動可能な次のステップを示す

rules: |
  - 出典（ドキュメント名・URL・該当箇所）を必ず明記
  - ナレッジに無い情報は「分かりません」と答える
  - 個人情報・機密情報を生成しない
  - 推測を述べる場合は「推測です」と明示
```

## 5. 設計パターン

### 5-1. 基本構成: RAG + ReAct

```
ユーザー質問 → クエリ展開 → ベクトル検索 → 関連文書取得 → LLM 回答生成 → 応答
                ↑               ↑              ↑           ↑
                LLM             ベクトルDB      Tool Use   Reflection
```

### 5-2. RAG の構成要素

- **チャンク化戦略**: 1チャンク = 500〜1000トークン、オーバーラップ 100トークン
- **埋め込みモデル**: OpenAI / Cohere / sentence-transformers
- **ベクトル DB**: Qdrant / Weaviate / pgvector / Chroma
- **検索戦略**: top-k (3〜10件) + リランキング

### 5-3. 会話履歴の扱い

| 方式 | 適合 |
|---|---|
| 単発 Q&A | 履歴なし、毎回独立 |
| 短期文脈 | 直近 3〜5 ターンを保持 |
| 長期記憶 | ユーザー別に重要事項を抽出して永続化 |

シンプルに始めるなら **単発 Q&A** か **短期文脈** から。

## 7. 評価データ

### 7-1. 質問パターン集（100件目安）

代表的な質問を集める:
- 簡単な事実検索（30件）
- 手順確認（20件）
- 判断補助（30件）
- ナレッジに無い質問（20件）→ 「分かりません」と答えるべき

### 7-2. 評価指標

| 指標 | 計測 | 目標 |
|---|---|---|
| 正答率 | 期待回答との一致 | 85%+ |
| 出典付与率 | 出典が含まれている | 100% |
| 「分からない」と答える率 | ナレッジ外の質問で適切に応答 | 95%+ |
| 平均応答時間 | 5秒以内 | |
| 1件あたりコスト | $0.02以下 | |

### 7-3. 評価方法

- **コードベース**: 出典の有無、応答時間、長さ
- **LLM as a judge**: 内容の正しさ・有用性
- **人間評価**: 抜き取りで継続的にチェック

## 8. ガードレール

アシスタント系では特に重要:

### 8-1. やってはいけないこと
- 個人情報の生成・推測
- 機密情報を社外に出す
- 法的助言の断定（「弁護士に相談」と誘導）
- 医療判断の断定（「専門医に相談」と誘導）

### 8-2. 実装

```python
# 入力フィルタ
if contains_pii(user_query):
    return "個人情報を含む質問にはお答えできません。"

# 出力フィルタ
if contains_confidential(response):
    return "申し訳ありません、その情報は提供できません。"

# プロンプトでも明示
SYSTEM_PROMPT = """
あなたは○○社のアシスタントです。
- 個人情報・機密情報を含む応答は禁止
- 法的・医療的な断定は禁止
- 不確かな情報は「分かりません」と答える
"""
```
