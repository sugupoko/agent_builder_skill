# ユースケースカタログ（LayerX 流）

> 出典: [LayerX「AIエージェント機能を継続的に生み出すPM」](https://zenn.dev/layerx/articles/b9272c63152826)

「**ユースケースを30件くらい列挙し、ニーズを把握しているPMをby nameで記載**する」のがLayerX流。
開発時に「想定しているユーザーニーズ、UX、プロセス上どんな機能だと成功になるか」を確認できる。

---

## なぜ必要か

エージェント開発で「これは作って意味があるのか？」を毎回判断するのは難しい。
ユースケースカタログがあると:

- どれから着手するか優先度判断に使える
- ステークホルダー間で認識を揃えられる
- 後で別エージェントを派生させるときの素材になる
- 「ニーズの証明」と「開発しやすさ」を分けて考えられる

---

## 作り方

### Step 1: 業務担当者・想定ユーザーをリストアップ

```
- 経営企画部の田中さん（業界ウォッチ）
- 経理部の佐藤さん（申請レビュー）
- マーケ部の鈴木さん（競合動向）
- 法務部の高橋さん（規制動向）
...
```

### Step 2: 各ユーザーが「困っている時間」をヒアリング

「今、AI じゃなくて手作業でやっている、時間がかかっていることは？」

```
- 田中さん: 毎週月曜の業界週次レポート作成に3時間
- 佐藤さん: 申請書のルール違反チェックに毎日30分×N人
- 鈴木さん: 競合の新製品リリースを Twitter/Web で追う
- 高橋さん: 業界ガイドライン改定の追跡
```

### Step 3: カタログ化

| # | ユースケース | 想定ユーザー | 想定頻度 | 価値 | 詳しい人 | 優先度 |
|---|---|---|---|---|---|---|
| 1 | 業界週次レポート自動生成 | 経営企画 | 週1 | 3h → 30min | 田中 | A |
| 2 | 申請書のルール違反チェック | 経理 | 申請毎 | 30min → 1min | 佐藤 | A |
| 3 | 競合の新製品ウォッチ | マーケ | 日次 | 1h → 5min | 鈴木 | B |
| 4 | 業界ガイドライン改定の追跡 | 法務 | 月1 | 2h → 15min | 高橋 | B |
| 5 | カスタマーサポートのFAQ自動応答 | CS | 都度 | 50%自動化 | 山田 | C |
| 6 | 社内ドキュメント検索 | 全社 | 都度 | 検索精度 | 中村 | B |
| 7 | 議事録から要約・TODO抽出 | 全社 | 会議後 | 30min → 5min | 吉田 | C |
| 8 | 営業資料のAIレビュー | 営業 | 提案毎 | 品質均一化 | 松本 | C |
| 9 | 採用書類の一次スクリーニング | HR | 応募毎 | 1h → 5min | 木村 | C |
| 10 | 技術文献のサーベイ | R&D | 月1 | 8h → 1h | 林 | B |
| ... | ... | ... | ... | ... | ... | ... |

優先度の判断軸:
- **A**: 時間削減効果が大きい × 技術的に着手しやすい
- **B**: どちらかが高い
- **C**: 探索的（やってみないと分からない）

## カタログの活用法

### 1. 着手順の決定

LayerX の指針: **エンジニア主導で「開発しやすい/技術的に突破しやすい」トピックから着手**。

最初から「最大価値」を狙うと頓挫しやすい。**動くものを作って評価サイクルを回す**ためには、技術突破しやすいケースから始めて、段階的に難しいケースへ。

### 2. 共通基盤の発見

カタログを並べると、複数ユースケースで共通して使う機能が見える:

```
ユースケース1, 3, 4: 外部RSS/Webから情報収集 → 共通の fetch ノード
ユースケース2, 9: 文書からルール違反検出 → 共通の review ノード
ユースケース6, 10: 検索クエリ生成と結果ランキング → 共通の retrieval ノード
```

→ 「**共通モジュール化** → 個別エージェントは薄く」の方針が立てられる。

### 3. ユースケース間の派生

最初のエージェント（ユースケース1）が動いたら、横展開しやすい:

```
ユースケース1（業界週次） で作ったエージェントを使い回す
↓ YAML だけ書き換える
ユースケース3（競合ウォッチ）→ 1日で動く
ユースケース4（規制追跡）→ 1日で動く
```

YAML 駆動の設計（`reference/theme_yaml_template.yaml`）が活きる。

## カタログの維持

### 月次でレビュー

- 完了したユースケースに「実績値」を書く
  - 例: 「3h → 30min」が予想 → 実際は「3h → 15min」だった
- 新しいユースケースを追加（業務担当からのフィードバック）
- 優先度の見直し（運用しながら判明した事情を反映）

### 「やらない」も記録する

カタログを作ると「全部やりたくなる」が、リソースは有限。
**やらないと決めたユースケース**にも理由を書いて残す:

```
| ユースケース | やらない理由 |
|---|---|
| AI 採用面接 | 法的リスク・倫理リスクが高い |
| 自動契約書生成 | 法務責任の所在不明確 |
| 顧客への自動返信 | ブランド毀損リスク |
```

これが「**捨てた選択肢の文書化**」となり、後で「あれ？やらないんだっけ？」と再検討する時に役立つ。

## カタログ ➜ エージェント設計への接続

特定のユースケースを実装する段で、`agent-discover` スキルに進む:

```
カタログから着手案件を選ぶ
  ↓
hearing_sheet_<種別>.md を印刷して詳細ヒアリング
  ↓
/agent-discover を実行 → spec.md 初版
  ↓
/agent-decompose → /agent-prototype → /agent-eval → /agent-deploy
```

カタログ全体は workspace/ の **プロジェクト共通** な位置（`workspace/usecase_catalog.md`）に置き、
各エージェントは workspace/<project>/ の中で個別に管理。

## まとめ

- **ユースケースを30件くらい列挙、by name で詳しい人を記載**
- **エンジニア主導で技術突破しやすいケースから着手**
- **共通基盤を発見**して個別エージェントを薄く保つ
- **やらないユースケース**も記録する
- 月次レビューで実績反映・新規追加・優先度見直し
