## 1. 業務理解（A. 業務）

### 1-1. 目的（富田）
- 即納が看板 → **一次回答スピードが命**。5 分遅れたら競合へ流出
- 件数 × 品質 × 速度の 3 点で勝負
- オペ 1 名 = 1 日 60〜100 件 / 1 件平均 6 分 / 月間 6 万件規模
- やめたら困る人: **営業（受注機会損失）／ 顧客（設計が止まる）／ 経理（請求滞留）**
- 手作業時間: オペ 1 名あたり 1 日 6〜10 時間

### 1-2. 業務ステップ（9 ステップ）
```
1. 問い合わせメール / チャットを受信（Salesforce / Outlook / Web チャット）
2. 内容を読み、何のカテゴリか判断（在庫? 技術? クレーム? 請求?）
3. 型番・注文番号を本文から抽出（複数あることも多い）
4. 必要な情報を社内 DB で検索（在庫 / 価格 / 出荷 / CAD ライブラリ）
5. 不足情報があれば顧客に逆質問（使用環境、希望納期、代替可否）
6. 回答文を作成（敬体、定型句、社内用語禁止）
7. クレームの匂いがあれば上長エスカレーション
8. 送信（Salesforce 経由、テンプレに差し込み）
9. ケース履歴をクローズ
```

→ **エージェント化したい範囲**: 1〜6 のうち特に **3（型番抽出）／ 4（DB 引き当て）／ 6（回答ドラフト）**。
→ **必ず人間が行う**: 8（送信）、7 のエスカレーション最終判断。LLM は「兆候フラグ」までを担当。

### 1-3. 「人の判断スキル」（中島の暗黙知 → 言語化）

ベテランの差は 3 つに集約される:

1. **型番の意味を読む力**
   - 例: `SHA-M6-20-N` → SHA がカテゴリ、M6 が呼びネジ、20 が長さ、N が表面処理（ニッケル）
   - 新人はこの分解ができないため、検索が遅い・取り違えが起きる

2. **裏の意図を察する力**
   - 「○○の代替品ある？」 → 半分のケースは「現品が廃番になっている」確認が真の意図
   - 「先回りして廃番情報＋後継品を提示」が正解

3. **クレームの種を嗅ぎ分ける力**
   - キーワード: 「困っています」「至急」「何度目ですか」「前回も」「結局」「まだ」
   - 文体パターン: 文頭が「お世話になっております」ではなく **いきなり用件から始まる**
   - これらの兆候が出たら、機械的応答禁止 → お詫び + 担当者引き継ぎ

### 1-4. editorial.persona の素材（重要）

```
1. 型番のプレフィックス・桁数・末尾オプションから、カテゴリと寸法とオプションを推定
2. 「代替品」「廃番」「後継」を聞かれたら、まず現品の生産状況を確認
3. 材質・耐熱・精度の質問は、単独回答せず使用環境を逆質問する
4. クレーム匂いキーワード（'至急', '何度目', '困って', 'まだ', '結局'）を検出したら、
   回答ドラフトの先頭に「お詫び一言 + 担当者引き継ぎフラグ」を挿入
5. 「とりあえず急ぎ」と書かれた場合、希望納期を必ず聞き返す
6. 注文番号と型番が混在しているメールは、両方とも処理対象として全部抽出する
```

→ これがそのまま `spec.md §5 編集者の視点` に入る。

### 1-5. 新人ペインポイント（高橋）
- 型番の打ち間違い（長文字列）
- 曖昧メール（「先日の件、どうなりましたか？」）から意図を読む
- 「工場の都合」など内部用語に頼らない言い換えに迷う

## 2. 入出力（B.）

### 2-1. 入力
- 種類: 自然言語（メール / Web チャット転記）
- 形式: プレーンテキスト + 引用履歴。たまに添付（PDF 図面 / Excel 見積）
- サイズ: 1 件 200〜2,000 字、引用込み最大 5,000 字
- 流入: 営業時間中、1 分に 1〜3 件
- ソース: Salesforce ケース / Outlook / Web フォーム
- 言語: JP 95% / EN 5%
- 顧客カテゴリ: 設計者 40% / 購買 30% / 製造現場 20% / 経理 10%

### 2-2. 出力
- 種類: (1) 構造化メタ JSON / (2) 顧客向け本文ドラフト（敬体）/ (3) 内部メモ（使用 DB / 信頼度）
- 配信先: Salesforce ケース内「AI ドラフト」タブ
- 量: 本文 200〜800 字、メタ約 10 フィールド
- 受け取り手: オペ（一次） + クレーム時 SV

## 3. 成功基準と評価（C.）

### 3-1. 「うまくいった」の定義（富田）
1. オペの応対時間が **30% 短縮**: 6 分 → 4 分
2. 新人の独り立ち期間が **半減**: 3 ヶ月 → 1.5 ヶ月
3. 誤回答による再問い合わせ **悪化させない**（絶対条件）

### 3-2. 評価データ計画
合成データ 20 件をシード（カテゴリ別）:

| カテゴリ | 件数 | 良い出力 | 悪い出力 |
|---|---|---|---|
| 在庫・納期確認 | 4 | 型番全部抽出、在庫数と納期併記、廃番なら代替提示 | 抽出漏れ、片方だけ、廃番見逃し |
| 技術質問（材質/精度）| 4 | 使用環境を逆質問、断定しない | 単独回答で断定 |
| 代替品問い合わせ | 3 | 廃番確認、後継 + 類似品を理由付きで | 「同等品です」だけ |
| 配送状況 | 3 | 注文番号特定、追跡 URL、遅延理由 | 注文番号取り違え、遅延を隠す |
| クレーム | 3 | お詫び一言、引き継ぎフラグ、即時返信 | 機械的応答、エスカレ漏れ |
| 請求・経理 | 3 | 担当部署引き継ぎ明記、自分で答えない | 推測で答える、誤った金額 |

### 3-3. 評価軸
- **コードベース**: 型番抽出再現率、必須テンプレ語含有率、文字数レンジ
- **LLM as a judge**: 内容妥当性 / トーン適切性 / 逆質問適切性
- **人間評価**: 月次で SV 2 名がランダム 20 件を 5 段階評価

## 4. 制約（D.）

### 4-1. コスト
- 1 件: $0.05 以下（Sonnet + Haiku 使い分け前提）
- 月: $3,000 上限（月 6 万件想定）
- 軽量モード: あり（Haiku のみ、定型ケース）

### 4-2. 速度
- 受信 → ドラフト: 30 秒以内
- リアルタイム必須

### 4-3. セキュリティ
- 機密度: 社内限 + 顧客個人情報含む
- LLM API 送信: OK（NDA で許諾済想定）。ただし氏名・電話・住所はマスキング
- ログに残す: 入力ハッシュ、出力本文、使用 DB、トークン消費量
- ログに残さない: 顧客氏名生データ、電話番号、住所
- 監査: 月次でランダム 100 件をマスキング前後で照合

### 4-4. 依存
- Salesforce Service Cloud / 在庫 DB（PostgreSQL） / 価格 DB（ERP API） / CAD ライブラリ / 認証 SSO（Okta）

## 5. 運用（E.）

### 5-1. トリガー
- イベント駆動: SF ケース新規作成 webhook、Outlook 受信箱 polling
- リトライ: 3 回（指数バックオフ）→ 失敗フラグ

### 5-2. レビュー
- 必須: オペが本文確認 → 編集 → 送信
- クレーム時: SV へ Slack 通知（ドラフト + 元メール）
- CRM に「AI 生成ドラフト使用」フラグを残す

### 5-3. 変更管理
- カテゴリ閾値・キーワード・テンプレ: YAML、CS センター長が直接編集
- 型番正規表現: SE レビュー必要
- LLM プロンプト変更: SE + センター長合議

### 5-4. メンテナンス
- 運用主体: CS 部 運用担当 2 名
- 監視: Datadog（レイテンシ・エラー率・コスト）
- 月次改善会: ベテランオペ 2 名 + SE 1 名

## 6. リスク（F.）

| # | リスク | 重大度 | フォールバック |
|---|---|---|---|
| 1 | 誤った納期 / 在庫情報を顧客に送信 | 致命 | DB 引き当て値だけ採用、LLM の数字捏造禁止 |

| 4 | コスト爆発 | 中 | recursion_limit、トークン上限、月予算アラート |
| 5 | LLM API ダウン | 中 | テンプレ定型 + 「AI 生成失敗」フラグ |
| 6 | DB ダウン | 中 | 「在庫情報取得失敗」テンプレ + 内部通知 |
