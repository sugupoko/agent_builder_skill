## 3. 判定（種別の特定）

エージェント種別: **複合型**
- アシスタント系（CS 一次対応の補助）が主軸
- 抽出系（型番・注文番号・添付メタの抽出）が前段で必須
- レビュー系（クレーム匂い検出・トーン判定）が品質ガード

→ ヒアリングシートとしては `hearing_sheet_assistant.md` をベースに、`hearing_sheet_extraction.md` の項目（抽出精度・正規表現）を組み合わせる必要がある。本案件は既にヒアリングが完了しているため、シート未記入のまま spec.md を起こしている。

## 4. 編集者ペルソナ（最重要）

ベテランオペ「中島」の暗黙知を 6 項目に言語化:

1. 型番のプレフィックス・桁数・末尾オプションから仕様推定
2. 「代替品/廃番/後継」は現品の生産状況を先回りで確認
3. 材質・耐熱・精度は単独回答せず逆質問
4. クレーム匂いキーワード検出時はお詫び + 引き継ぎフラグ
5. 「とりあえず急ぎ」には希望納期を必ず聞き返す
6. 注文番号と型番が混在したら両方とも全件抽出

→ `spec.md §4 編集者ペルソナ` および `§5 編集者の視点` に転記。

## 8. 設計上の所見（次スキルへのメモ）

### Decompose 担当への申し送り

- **採用パターン候補**: Tool Use + Reflection（Multi-Agent / Planning は v1 不要）
- **推奨ノード分割**（仮）:
  ```
  1. preprocess（マスキング、言語判定、添付検出）
  2. extract（型番・注文番号 regex 抽出 + LLM 補完）
  3. classify（カテゴリ + 緊急度 + クレーム匂い）
  4. retrieve（在庫 / 価格 / 出荷 / 廃番 DB 引き当て、Tool Use）
  5. draft（回答本文起草、Sonnet）
  6. reflect（数字捏造チェック / 必須語含有 / クレーム時の引き継ぎ確認）
  7. assemble（メタ JSON + 本文 + 内部メモ）
  ```
- **モデル使い分け候補**:
  - Sonnet 4.6: classify / draft / reflect
  - Haiku 4.5: extract（regex の補完）/ 定型ケース起草（軽量モード）
- **YAML 駆動範囲**: カテゴリキーワード辞書 / クレーム匂いキーワード / テンプレ文言 / 型番正規表現

### Prototype 担当への申し送り

- DB は **モック（YAML フィクスチャ）** で開始。実 DB 接続は deploy フェーズ
- マスキング層は**LLM 送信前の関数で実装**（顧客氏名・電話・住所）
- Salesforce 連携は v1 ではスコープ外。CLI 入力 → JSON 出力の最小スクリプトで OK

### Eval 担当への申し送り

- 評価軸 10 項目（spec.md §9）のうち、目標値が「最重要」なのは **#1 型番抽出 95%**、**#3 クレーム再現率 95%**、**#10 誤回答悪化なし**
- LLM Judge プロンプトは「ベテランオペ中島」視点で書く（5 次元採点）

## 10. 既存リファレンス実装との関係

`workspace/cs_triage_agent/v1/` に同一ドメインのリファレンス実装が存在する（9 ノード LangGraph、LLM Judge 評価、運用設計まで揃った成熟実装）。本プロジェクト v3 は **同じヒアリング前提を再投入して discover 〜 deploy を再走行** する位置付け。リファレンス実装と比較しながら、改善点・差分（特にノード分割や Reflection の入れ方）を `decompose` 以降で精査するとよい。

ただしリファレンス実装の意思決定をそのままなぞると discover の意味がないので、本 v1 は spec.md を白紙からヒアリング駆動で組み上げた。**v2 で v1（リファレンス）と統合 / 比較**するのが自然な流れ。
