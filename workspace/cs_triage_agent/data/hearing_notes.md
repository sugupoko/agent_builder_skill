## A. 業務の理解

### A-1. 業務の目的（富田 → 全員共通）

> **富田**: 「うちは即納が看板だから、問い合わせに対する一次回答スピードが命です。お客さんは設計者や購買担当で、平日の日中に『型番の在庫いくつ?』『この代替品ある?』みたいに次々来る。これに5分でも遅れると、競合の○○商事に流れる。だから件数 × 品質 × 速度の3点で勝つしかない。」
>
> 「今は1人あたり1日 60〜100件の問い合わせメール+チャットを捌いてる。1件平均6分。月間 6万件規模。やめたら受注が減る、シンプルに。」

- **やめたら困る人**: 営業 (一次対応が遅れて受注機会損失)、お客様 (設計が止まる)、経理 (請求関連の問い合わせが滞る)
- **手作業時間**: オペ1名あたり 1日 6〜10時間、月160時間の純粋な応対

### A-2. 業務ステップ分解

```
1. 問い合わせメール/チャットを受信 (Salesforce / Outlook / Web チャット)
2. 内容を読み、何のカテゴリか判断 (在庫? 技術? クレーム? 請求?)
3. 型番・注文番号を本文から抽出 (本文中に複数あることも多い)
4. 必要な情報を社内DBで検索 (在庫DB / 価格DB / 出荷DB / CADライブラリ)
5. 不足情報があれば顧客に逆質問 (使用環境、希望納期、代替可否)
6. 回答文を作成 (敬体、定型句あり、社内用語禁止)
7. クレームの匂いがあれば上長エスカレーション
8. 送信 (Salesforce 経由、テンプレに差し込み)
9. ケース履歴をクローズ
```

→ **エージェント化したい範囲**: 1〜6 のうち、特に **3 (型番抽出)、4 (DB引き当て)、6 (回答ドラフト)**。最終送信 (8) は必ず人間。エスカレーション判断 (7) は LLM で「兆候フラグ」を出すが、決定はオペ。

### A-3. 「人の判断スキル」の所在 (中島 から引き出す)

> **中島**: 「経験が効くのは大きく3つ。**型番の意味を読む力**、**裏の意図を察する力**、**クレームの種を嗅ぎ分ける力**ですね。」
>
> 「型番の意味を読む力 — うちの型番は接頭辞でカテゴリ、桁数で寸法、末尾でオプションが分かるんです。新人は『SHA-M6-20-N』って見たときに、SHA がカテゴリで M6 が呼びネジで20が長さで N が表面処理、ってのがピンと来ない。」
>
> 「裏の意図 — 『○○の代替品ある?』って聞かれたとき、新人は素直に同等品を返すんですが、本当は『今の型番が廃番になっててもう手に入らない』を確認したいケースが半分。先回りして『この型番は廃番予定です。後継は△△です』と言うと喜ばれる。」
>
> 「材質質問は逆質問が鉄則。『SUS304とSUS316の違いは?』だけ答えると、お客が間違った選択をする。**使用温度・薬品環境・コスト要件**を聞いてから答えないと、半年後にクレームになって返ってくる。」
>
> 「クレームの匂い — メール本文に『困っています』『至急』『何度目ですか』『前回も』が入ってたら、もう半分は不満。文頭が『お世話になっております』じゃなくいきなり用件から始まる場合も要注意。」

#### 言語化された判断スキル (editorial.persona の素材)

```
1. 型番のプレフィックス・桁数・末尾オプションから、カテゴリと寸法とオプションを推定
2. 「代替品」「廃番」「後継」を聞かれたら、まず現品の生産状況を確認
3. 材質・耐熱・精度の質問は、単独回答せず使用環境を逆質問する
4. クレーム匂いキーワード ('至急', '何度目', '困って', 'まだ', '結局') を検出したら、
   回答ドラフトの先頭に「お詫び一言 + 担当者引き継ぎフラグ」を挿入
5. 「とりあえず急ぎ」と書かれた場合、希望納期を必ず聞き返す
6. 注文番号と型番が混在しているメールは、両方とも処理対象として全部抽出する
```

### A-4. 新人ペインポイント (高橋 から)

> **高橋**: 「私が一番困るのは、型番が長くて打ち間違えること、あとお客様のメールから何が知りたいのか読み取れない時です。『先日の件、どうなりましたか?』だけのメールが来て、ケース履歴を見にいくのに5分かかったり…」
>
> 「あと、納期回答で『工場の都合』とか書きたくないんですけど、何て書けば失礼にならないか毎回中島さんに聞いてます。テンプレートはあるんですが、ケースバイケースで使い分けが難しい。」

## B. 入力と出力

### B-1. 入力

| 項目 | 値 |
|---|---|
| 種類 | 自然言語 (メール / Web チャット転記) |
| 形式 | プレーンテキスト + 引用履歴。たまに添付 (PDF図面、Excel見積) |
| サイズ | 1件 200〜2000文字。引用込みで最大 5000文字 |
| 更新頻度 | 営業時間中、1分に1〜3件流入 |
| ソース | Salesforce ケース / Outlook 専用受信箱 / Web フォーム |
| 言語 | 日本語 95%、英語 5% (海外調達担当顧客) |
| 顧客カテゴリ | 設計者 (40%)、購買 (30%)、製造現場 (20%)、経理 (10%) |

### B-2. 出力

| 項目 | 値 |
|---|---|
| 種類 | 回答ドラフト (Markdown / プレーンテキスト) + メタ情報 (カテゴリ / 緊急度 / 抽出型番) |
| 配信先 | Salesforce のケース内『AI ドラフト』タブ。オペが手動で本文へコピー or 編集 |
| 形式 | (1) 構造化メタ JSON (2) 顧客向け本文ドラフト (敬体) (3) 内部メモ (使用したDB / 信頼度) |
| 量 | 本文 200〜800文字。メタ JSON 10フィールド程度 |
| 受け取る人 | オペレーター (一次)。クレーム匂い検出時はスーパーバイザーにも通知 |

## C. 成功基準と評価

### C-1. 「うまくいった」の定義

> **富田**: 「現場の感覚的な KPI は3つです。」
> 1. **オペの応対時間が30%短縮**: 1件あたり 6分 → 4分
> 2. **新人の独り立ち期間短縮**: 3ヶ月 → 1.5ヶ月
> 3. **誤回答による再問い合わせ件数 ゼロ維持** (これは絶対条件、悪化したら導入失敗)

### C-2. 評価データ

合成データで開始。種類別に各3件以上、合計 20件をシードに:

| カテゴリ | 件数 | 良い出力の特徴 | 悪い出力の特徴 |
|---|---|---|---|
| 在庫・納期確認 | 4 | 型番を全部抽出、在庫数と納期を併記、廃番なら代替提示 | 型番抽出漏れ、納期だけ・在庫だけ、廃番見逃し |
| 技術質問 (材質/精度) | 4 | 使用環境を逆質問、断定しない | 単独回答で断定、推測なのに断定口調 |
| 代替品問い合わせ | 3 | 廃番確認、後継+類似品を理由付きで | 単に「同等品です」とだけ返す |
| 配送状況 | 3 | 注文番号特定、追跡URL、遅延理由 | 注文番号取り違え、遅延を隠す |
| クレーム | 3 | お詫び一言、担当者引き継ぎフラグ、即時返信 | 機械的応答、エスカレーション漏れ |
| 請求・経理 | 3 | 担当部署引き継ぎ明記、自分で答えない | 推測で答える、誤った金額を提示 |

### C-3. 評価方法

- **コードベース**: 型番抽出精度 (正規表現 / 期待型番との完全一致率)、必須テンプレ語の含有 (「お世話になっております」「ご確認のほど」等)、文字数レンジ
- **LLM as a judge**: 内容の妥当性、トーンの適切さ、逆質問の適切性
- **人間評価**: 月次でスーパーバイザー2名が抜き取り20件を5段階評価

## D. 制約と非機能要件

### D-1. コスト
- 1件あたり予算: **$0.05以下** (Sonnet+Haiku 使い分け前提)
- 月間上限: **$3,000** (月6万件想定)
- 軽量モード: あり (Haiku のみ、簡易質問用)

### D-2. 速度
- 受信から回答ドラフト生成まで: **30秒以内** (オペが体感で待てるライン)
- バッチ処理は不可、リアルタイム必須

### D-3. セキュリティ
- データ機密度: **社内限+顧客個人情報含む** (氏名・会社名・電話番号・注文履歴)
- 外部 LLM API 送信は **OK** (現行 NDA で API 利用権限あり想定)。ただし氏名・電話番号は送信前にマスキング
- ログに残す: 入力ハッシュ・出力本文・使用DB・トークン消費量
- ログに残さない: 顧客氏名生データ、電話番号、住所
- 監査: 月次でランダム100件をマスキング前後で照合

### D-4. 依存関係
- Salesforce (Service Cloud) 連携必須
- 在庫DB (社内 PostgreSQL)、価格DB (基幹システム ERP の API)
- CADライブラリ (社内ファイルサーバ、URL返すだけでOK)
- 認証: 社内 SSO (Okta) 想定

## E. 運用と継続性

### E-1. トリガー
- イベント駆動: Salesforce ケース新規作成 / メール受信 webhook
- リトライ: 失敗時3回、それでもダメならオペに「AIドラフト失敗」通知

### E-2. レビュー工程
- **必須**: オペが必ず本文を確認 → 編集 → 送信。AIが顧客に直接送らない
- クレーム匂い検出時: スーパーバイザーへ Slack 通知 (ドラフト+元メール)
- 送信前に「AI生成ドラフトを使用」フラグを CRM に残す (監査用)

### E-3. 変更管理
- カテゴリ閾値・キーワード・テンプレ文言: YAML で運用、CSセンター長が直接編集
- 型番正規表現の追加: SE が変更レビュー必要
- LLM プロンプト変更: SE + センター長の合議

### E-4. メンテナンス
- 運用主体: CS 部 (運用担当2名)
- 監視: Datadog でレイテンシ・エラー率・コスト
- 月次改善会: ベテラン2名 + SE 1名で「ドラフトが微妙だったケース」をレビュー

## F. リスクと失敗モード

### F-1. 重大度順

| # | リスク | 重大度 | 顕在化シナリオ |
|---|---|---|---|
| 1 | 誤った納期/在庫情報を顧客に送信 | 致命 | DBから取った値が古い、または抽出型番が間違ってる |
| 2 | クレーム見逃しで顧客大激怒 | 高 | クレーム匂い検出失敗 → 機械的回答 → 二次クレーム |
| 3 | 顧客個人情報の外部漏えい | 高 | マスキング漏れで LLM API に氏名・電話を送信 |
| 4 | コスト爆発 | 中 | 無限ループ・大量メール時 |
| 5 | システム停止で全停止 | 中 | LLM API ダウン、SF 連携不通 |

### F-2. フォールバック
- LLM ダウン: テンプレ定型文 + 「AI 生成失敗、手動対応してください」フラグ
- DB ダウン: 「在庫情報取得失敗。担当より折り返しご連絡」テンプレ
- 抽出型番が DB にない: 「在庫情報を確認中。お時間いただきます」テンプレ + 内部通知
