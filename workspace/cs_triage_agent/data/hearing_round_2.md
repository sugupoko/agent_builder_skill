## A. 業務カテゴリと判定ルール（§16-1, §16-3 関連）

### A-1. 多言語対応の判定ロジック（§16-1 / 項目 2）

> **富田**: v1 は日本語 95% / 英語 5% で、英語は『英語担当に転送』テンプレで運用するんでしたね。判定はどうしてますか？

> **中島**: 今は本文に「ひらがな・カタカナがあるか」で雑に判定してます。記号や型番だけだと判定が当たらない。

**合意**:
- v3 では `langdetect` ライブラリ（オフライン）を導入。Claude にやらせるとコスト増 + 精度ばらつき。
- 「ja=日本語確率 0.6 以上」を閾値、それ以外は en/other に倒す。
- ja とも en とも判定不能なケース（記号 + 型番のみ）は **「短文 + 型番あり」だったら ja として処理**（型番質問の慣習）。
- 実装は SE 工藤主管。

### A-2. 添付ファイルの扱い（§16-1 / 項目 3）

> **富田**: PDF 図面と Excel 見積はオペが目で見るしかないですよね？

> **中島**: 添付があるかどうかだけは検出して、「添付ファイルの内容は手動でご確認ください」と内部メモに残してくれると助かる。

**合意**:
- v1 のままで添付検出はあるので、`internal_memo` に **「添付あり: 手動確認をお願いします」** を必ず付与（PDF / Excel の中身は v3 でも処理しない）。
- v4 以降で OCR を検討するが、今は対象外。

### A-3. ケース履歴遡及（§16-1 / 項目 4）

> **富田**: 「先日の件、どうなりましたか？」は本当に困る。Salesforce のスレッド ID で 30 日分自動取得してほしい。

> **工藤**: スレッド ID から `Case` オブジェクトを参照して、過去 30 日のメール履歴を取得する API はあります。レイテンシは +1〜2 秒、件あたりトークン +500 程度。

**合意**:
- v3 で `lookup_case_history(case_id, days=30)` を新ツールとして追加。retrieve ノード内で「カテゴリが unknown または『先日の件』類似フレーズ検出時」のみ呼ぶ。
- 履歴は最大 5 件 / 1,000 文字に切り詰めて draft プロンプトに含める。
- ⚠️ 自動遡及は便利だが、ケース ID が誤ってる場合に他人の履歴を見る危険があるため、**取得した case_id と顧客メールアドレスの一致確認をコード側で行う**（工藤主管）。

### A-4. complaint × shipment の最終カテゴリ（§16-3 / 項目 16）

> **阿部（SV）**: クレーム調 + 注文番号がある場合、私の経験上 8 割は「主訴はクレーム、副要素として配送」です。SV 通知は必ず欲しい。

> **中島**: でも shipment ナレッジでも DB 引き当ては必要なので、両方フラグが立つのが理想ですね。

**合意**:
- カテゴリは **complaint** を優先（complaint_smell=True なら category も complaint に強制）。
- ただし retrieve ノードでは **complaint_smell=True かつ order_no 抽出済みなら lookup_shipment も並走**（追跡情報を内部メモに）。
- v3 で classify ノードの prompt に「complaint_smell が True なら category=complaint を最優先」と明示。

### A-5. urgency ラベルの定義（§16-3 / 項目 14）

> **富田**: low / normal / high の 3 段階で運用してますが、感覚で振っていて統一基準がありません。

> **阿部**: SV としては「いつまでに対応するか」で決めたい。

**合意（運用ルール化）**:
- **high**: クレーム匂い検出 OR 「至急」「今日中」「明日まで」「製造ライン停止」キーワード OR 配送遅延 24h 超過
- **normal**: 上記に該当せず、24 時間以内に応対希望
- **low**: 「お時間あるとき」「いずれ」など緊急性のない表現
- LLM プロンプトに上記ルールを明示。eval メタデータも同基準に揃える（一致率向上が見込める）。

### A-6. apology 文言の運用基準（§16-3 / 項目 15、項目 21）

> **中島**: complaint=True で「申し訳ございません」、False で「お手数」だけ、というのは v2 の規則として理解してます。ただ、配送遅延ぎみの問い合わせ（complaint=False だが客は不安）でも「ご不便かと存じますが」は入れたい。

> **阿部**: 共感表現は SV 視点でも歓迎です。お詫びの「具体的な責任表明」（申し訳・お詫び・ご迷惑）は禁止、共感（ご不便・ご心配・お手数）は OK で分けるのが良いと思います。

**合意（v3 ルール）**:
- 禁止語（complaint=False 時）: 「申し訳ございません」「お詫び申し上げます」「ご迷惑をおかけし」
- 許容語（complaint=False でも）: 「お手数ですが」「ご不便かと存じます」「ご心配をおかけしますが」
- v2 の rules を v3 で更新。eval スコアの apology_kws も更新（既に「お手数」は除外、「ご不便・ご心配」は応対品質向上の観点から検出から除外）。

### A-7. DB 値積極記載のカテゴリ別ポリシー（§16-3 / 項目 18, 項目 20）

> **富田**: 在庫数を本文に書くか書かないかは慎重に判断したい。

> **中島**: 私の経験では「100 個以上ある」「数日中に出荷可」のレベル感は伝えてよい。**正確な在庫数（例: 1234 個）は書かない**ほうが業務的に安全（顧客が「あるなら全部買う」となる）。

> **阿部**: 納期日数は数値で書くべき。「14 営業日」のような単位明示が必要。

**合意（カテゴリ別ポリシー）**:

| 情報 | inventory | shipment | alternative | tech | billing | complaint | cad |
|---|---|---|---|---|---|---|---|
| 在庫数 | **レベル感**（充足/部分/欠品）| - | レベル感 | - | - | レベル感 | - |
| 納期日数 | **数値（営業日明示）** | - | 数値 | - | - | 数値 | - |
| 配達予定日 | - | **YYYY-MM-DD で明示** | - | - | - | 明示 | - |
| 追跡番号 | - | **業者名 + 追跡 URL を提示**（追跡番号は内部メモのみ）| - | - | - | URL のみ | - |
| 価格 | **税抜単価を数値で**（取引先別は customer_id 連携後）| - | - | - | - | - | - |
| 廃番情報 | - | - | **後継品 + 理由必須** | - | - | - | - |

→ v2 の「全件で DB 値積極記載」をやめ、**情報の種別ごとに level / number / date / url の使い分け** をプロンプトで指示。

### A-8. shipment / billing の内部用語禁止辞書（項目 23）

> **中島**: 「資材」「対応中」「上が言うには」「弊社内事情」あたりは内部用語ですね。Judge も指摘してくれてました。

**合意**:
- v3 の YAML に `forbidden_internal_phrases` リストを追加し、`reflect_node` でハードチェック:
  - 「工場の都合」「うちのシステム」「上が言うには」「弊社内事情」「資材の入荷」「対応中」「社内検討中」「営業に確認」
- 検出されたら reflect_pass=False、再起草を 1 回ループ。

### A-9. rule-based classify のタイブレーク（§16-2 / 項目 10）

> **工藤**: dry-run（軽量モード起動の前段判定）でタイブレークが起きる場合、優先順位を YAML に定義しましょう。

**合意**:
- YAML に `category_priority`: `[complaint, alternative, shipment, billing, tech, cad, inventory, other]` を定義（complaint > alternative の理由は SV 通知最優先）。
- v3 で適用。

### A-10. draft の missing_info の表示先（§16-2 / 項目 11）

> **富田**: Salesforce 側の AI ドラフトタブの「逆質問候補」リストに表示したい。

> **工藤**: SF カスタムオブジェクト `AI_Draft__c` に `missing_info_json__c`（Long Text 32k）を新設します。

**合意**:
- SF 側に専用フィールドを新設。オペが見る画面では「逆質問候補」セクションとして表示。
- 実装は工藤主管、deploy フェーズでの作業。

## B. 個人情報・セキュリティ（§16-2, §16-3 関連）

### B-1. PII マスキングの厳密さ（§16-3 / 項目 13）

> **大野（法務）**: 顧客氏名は **完全マスク**でお願いします。「田○様」のような部分マスクは GDPR / 改正個情法的にグレーで、内部統制監査でも指摘されます。

> **工藤**: 完全マスク（`[NAME_001]`）は v1 の現行実装ですね。

**合意**:
- 顧客氏名は **完全マスク** で確定。LLM API には placeholder のみ送信。
- 復元（unmask）は assemble の最終段でのみ。

### B-2. PII マスキング誤検知の許容範囲（§16-2 / 項目 9）

> **大野**: 誤検知（False Positive、PII でないものを PII としてマスク）は許容しますが、**漏れ（False Negative、PII を見逃して LLM API に流す）は絶対 NG**。

> **工藤**: v1 の電話 regex の貪欲マッチ罠（注文番号 ORD-2026-... の `0` 起点誤マッチ）は対症療法済みですが、英数字混在型番に対する耐性は要検証。

**合意**:
- **再現率 100% 必達**（漏れゼロ）、適合率は 80% 程度で OK。
- v3 で **PII マスキング regex の単体テスト 50+ 件** を整備（漏れケース 30 件 + 誤マスクケース 20 件）。
- 月次 PII 監査でランダム 100 件をマスキング前後照合（spec §10-3 で既定）。

### B-3. unmask 後の PII を本文に残すかのポリシー（項目 22）

> **大野**: AI ドラフトの最終本文には PII（氏名・電話など）が含まれます。これは元メールに含まれていた情報の再記載なので問題ない（顧客が自分で書いた情報）。ただし **draft prompt 段階では LLM に PII を渡さず、本文には placeholder のまま渡す**べき。

> **中島**: それでいい。本文の PII はオペが最終確認する直前に unmask されてれば十分。

**合意（重要な変更）**:
- v3 で draft プロンプトには `[NAME_001]` 等の placeholder 形式のまま本文を生成させる。
- assemble ノードで unmask して `customer_body` に書き戻す。
- LLM Judge は v3 から **マスク済み本文** を採点する（PII を Judge にも見せない）。

→ これは v2 の実装で「Judge コメント: 『PII (電話番号) を本文に記載』」で指摘されたケース 15 を解消する仕様変更。

### B-4. CAD URL の顧客送出可否（§16-3 / 項目 17）

> **大野**: 内部 CDN URL（`*.internal.example.com`）は **絶対に顧客に送らない**。社内サーバ DNS は機密相当です。
> 外部公開可能な署名付き短期 URL（CloudFront 等）が用意できれば送付可、ただし IT 部門との合意必要。

> **工藤**: 短期署名 URL は実装可能ですが、v3 のスコープ外（v4 検討）。当面は v2 の「担当者よりご案内」テンプレで運用。

**合意**:
- v2 の実装方針（本文では URL を出さず内部メモのみ）を継続。
- v4 以降で短期署名 URL を IT 部門と検討。

### B-5. dry-run / real LLM の運用境界（§16-3-A / 項目 19）

> **工藤**: PR レビュー / CI = dry-run、リリース判定 / シャドー = real。これで合意できますか？

> **富田**: 月次 eval は real でやりましょう。22 件で約 $1 なら全然 OK。

**合意（運用ルール）**:
- **CI（毎 push）**: dry-run、コード変更で構造が壊れていないかを確認
- **PR マージ前**: dry-run + 失敗ケース 5 件のみ real（コスト約 $0.15）
- **リリース判定**: real 全件 + LLM Judge 全件（コスト約 $0.6 / 月次）
- **シャドー運用**: real のみ（Judge は要時のみ）
- README / runbook に明記する（deploy フェーズ）。

## C. クレーム検出の閾値設計（§16-3 関連）

### C-1. クレーム匂い閾値（§16-3 / 項目 12）

> **阿部（SV）**: 過検知ライン: SV が日次で捌けるのは **20 件まで**。月 6 万件問い合わせなら、`(20 件/日 × 22 営業日) / 60,000 件 = 0.73%`。precision は **70% 程度**でも捌けます。
> 漏れは絶対 NG（recall 95%+ 必須）。

> **富田**: 現状の v2 でクレーム recall 100%, precision 75% は SV ライン的には許容範囲ですね。

**合意**:
- recall 95% 以上必達、precision 60% 以上目標（v2 の 75% は十分）。
- 月次でクレーム検出件数を集計、precision が 60% 切ったらキーワード辞書を見直し。
