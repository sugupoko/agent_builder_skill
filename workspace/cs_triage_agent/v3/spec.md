## 1. 目的

B2B 製造業（部品・工具・FA サプライ商社）のコールセンターに届く顧客問い合わせ（メール / Web チャット）を **トリアージし、回答ドラフトを生成する**。最終送信は必ずオペが行い、エージェントは「下書き屋 + 一次解析者」に徹する。1 件 6 分の応対時間を 4 分に短縮し、新人独り立ち期間を半減（3 ヶ月 → 1.5 ヶ月）させるのが現場 KPI。

## 2. エージェント種別

- [x] アシスタント系（CS 一次対応の補助）
- [x] 抽出系（型番・注文番号・添付メタの抽出）
- [x] レビュー系（クレーム匂い検出・トーン判定）
- [ ] リサーチ系
- [x] **複合**

## 3. 入力 / 出力

### 3-1. 入力

| 項目 | 値 |
|---|---|
| 種類 | 自然言語（メール本文 / Web チャット転記） |
| 形式 | プレーンテキスト + 引用履歴。たまに添付（PDF 図面・Excel 見積） |
| サイズ | 1 件 200〜2,000 文字。引用込みで最大 5,000 文字 |
| 更新頻度 | 営業時間中、1 分に 1〜3 件流入（ピーク時 月 6 万件規模） |
| ソース | Salesforce Service Cloud のケース / Outlook 専用受信箱 / Web フォーム |
| 言語 | 日本語 95% / 英語 5%（v1 は日本語のみ処理、英語は「英語担当へ転送」テンプレを返す） |
| 顧客カテゴリ | 設計者 40% / 購買 30% / 製造現場 20% / 経理 10% |

### 3-2. 出力

| 項目 | 値 |
|---|---|
| 種類 | (1) 構造化メタ JSON、(2) 顧客向け本文ドラフト（敬体）、(3) 内部メモ（使用 DB / 信頼度 / フラグ） |
| 配信先 | Salesforce ケース内「AI ドラフト」タブ。オペが手動で本文へコピー or 編集して送信 |
| 形式 | メタ JSON は約 10 フィールド（カテゴリ / 緊急度 / 抽出型番 / 注文番号 / 顧客タイプ / クレームフラグ / 信頼度 / 不足情報 / 引き継ぎ先 / 使用 DB） |
| 量 | 顧客向け本文 200〜800 文字。内部メモ 100〜300 文字 |
| 受け取る人 | オペ（一次）。クレーム匂い検出時はスーパーバイザーへ Slack 通知 |
| 同期/非同期 | 同期（受信から 30 秒以内にドラフトを CRM 上に表示） |

> ヒアリングで引き出したベテランオペ「中島（42 歳・勤続 11 年）」の判断スキルを、システムプロンプトに以下の人格で注入する：

```
あなたは中堅 B2B 部品商社のベテラン CS オペレータ。勤続 10 年超で、機械工学の独学知識を持ち、
材料力学・表面処理・公差まで一通り答えられる。新人が見抜けない「型番に隠れた仕様」「客が
本当に聞きたかったこと」「クレームに化ける兆候」を嗅ぎ取るのが得意。
あなたの役割は最終回答ではなく、若手オペが顧客に送る前の「下書き」を高速に仕上げること。
推測で断定せず、必要なら逆質問で深掘りし、不確実な情報には必ず「DB 引き当て中 / 要確認」
と但し書きを添える。クレーム匂いを 1 秒で察知し、お詫びの一言と引き継ぎフラグを置くのが
あなたの腕の見せどころ。
```

## 5. 編集者の視点

判断ロジックを 6 項目に言語化（システムプロンプト + ハードコードされたルール双方に反映）：

1. **型番のプレフィックス・桁数・末尾オプション** から、カテゴリ／呼び寸法／表面処理を推定する
2. 「代替品」「廃番」「後継」を聞かれたら、まず **現品の生産状況を確認** → 廃番なら後継品を理由付きで提示
3. 材質・耐熱・精度の質問は **単独回答で断定しない**。使用温度・薬品環境・コスト要件を逆質問してから答える
4. クレーム匂いキーワード（「至急」「何度目」「困って」「まだ」「結局」「前回も」）を検出したら、回答冒頭に **お詫び一言 + 担当者引き継ぎフラグ** を挿入
5. 「とりあえず急ぎ」と書かれた場合、**希望納期を必ず聞き返す**
6. 注文番号と型番が混在しているメールは、**両方とも全件抽出** して処理対象にする

## 6. 執筆・出力ルール

| カテゴリ | ルール |
|---|---|
| 必ず含める | 「お世話になっております」（冒頭）／「ご確認のほどよろしくお願いいたします」（締め）／自社署名プレースホルダ |
| 絶対に書かない | 社内用語（「工場の都合」「上が言うには」「うちのシステム」）／断定的な誤情報／値引き約束／顧客個人情報の生データ |
| 文体 | 敬体・丁寧語。一文 60 文字以内を目安に |
| 数字 | DB 引き当てた値のみ。LLM が「在庫数」「価格」を推測で書くことを禁止 |
| 不確実性の表現 | 「現在確認中です」「担当より折り返しご連絡します」を許容。決して「ありません」「できません」を断定しない |
| 引き継ぎ | 請求・経理・法務・規格証明系は「担当部署引き継ぎ明記」、自分で答えない |

## 7. データソース（社内システム連携）

| 名称 | 種類 | 目的 | 連携方式 | 備考 |
|---|---|---|---|---|
| 在庫 DB | 社内 PostgreSQL | 型番に対する在庫数・出荷可能日 | 直接 SQL（read-only ビュー） | リアルタイム |
| 価格 DB（ERP API） | 基幹システム | 型番に対する単価・取引先別価格 | REST API | 認証は SSO（Okta） |
| 出荷 DB | 社内 PostgreSQL | 注文番号に対する出荷状況・追跡番号 | 直接 SQL | リアルタイム |
| CAD ライブラリ | 社内ファイルサーバ | 型番に対する CAD/PDF 図面の URL | URL 文字列を返すだけ | OCR 等は v1 範囲外 |
| 廃番マスタ | 社内 PostgreSQL | 型番に対する廃番フラグ・後継品 | 直接 SQL | 月次更新 |
| Salesforce Service Cloud | CRM | ケース取得・ドラフト書き込み・履歴照会 | REST API（Salesforce Connect） | webhook トリガー |

> 本番接続は v1 では **モック**（YAML / JSON フィクスチャ）で代替。実 DB 接続は deploy フェーズで実装。

## 8. カテゴリ構成

| カテゴリ | 内容 | 期待ノード処理 |
|---|---|---|
| 在庫・納期確認 | 型番の在庫数・納期 | 型番抽出 → 在庫 DB 引き当て → 納期付与 |
| 技術質問 | 材質・耐熱・精度・寸法・互換性 | 逆質問テンプレ → 確認事項を列挙 |
| 代替品・廃番 | 「代替ある？」「後継は？」 | 廃番マスタ照会 → 後継提示 → 理由付与 |

| CAD・図面 | 「図面くれ」 | 型番 → CAD ライブラリ URL 返却 |
| 請求・経理 | 「請求書再発行」「金額違う」 | 経理部署引き継ぎテンプレ |

| その他 | 上記に当てはまらない／曖昧 | 「お問い合わせ確認中」テンプレ |

## 10. 制約

### 10-1. コスト
- 1 件あたり予算: **$0.05 以下**（Sonnet + Haiku 使い分け前提）
- 月間上限: **$3,000**（月 6 万件想定）
- 軽量モード: **必要**（Haiku のみで「在庫照会」「配送状況」など定型ケースを処理）

### 10-2. 速度
- 受信 → ドラフト表示: **30 秒以内**（p95）
- バッチ不可、リアルタイム必須
- LLM API レイテンシは Haiku 〜2s、Sonnet 〜5s を見込む

### 10-3. セキュリティ・コンプライアンス
- 機密度: **社内限 + 顧客個人情報含む**（氏名・会社名・電話・注文履歴）
- 外部 LLM API 利用: **OK**（NDA で許諾済の想定）。ただし **氏名・電話番号・住所は送信前にマスキング**
- ログに残す: 入力ハッシュ、出力本文、使用 DB、トークン消費量、信頼度
- ログに残さない: 顧客氏名・電話番号・住所の生データ
- 監査: 月次でランダム 100 件をマスキング前後で照合

### 10-4. 依存関係
- Salesforce Service Cloud（必須）
- 在庫 DB / 価格 DB / 出荷 DB（社内 PostgreSQL + ERP API）
- CAD ライブラリ（社内ファイルサーバ）
- 認証: 社内 SSO（Okta）

## 11. 運用条件

### 11-1. トリガー
- イベント駆動: Salesforce ケース新規作成 webhook ／ Outlook 受信箱の polling（5 秒間隔）
- リトライ: 失敗時 3 回（指数バックオフ）。それでもダメなら「AI ドラフト失敗」フラグでオペに通知

### 11-2. 配信前レビュー
- **必須**: オペが必ず本文を確認 → 編集 → 送信。AI が顧客に直接送らない
- クレーム匂い検出時: スーパーバイザーへ Slack 通知（ドラフト + 元メールのリンク）
- CRM に「AI 生成ドラフトを使用」フラグを残す（監査用）

### 11-3. 変更管理
- カテゴリ閾値・キーワード・テンプレ文言: **YAML で運用**、CS センター長が直接編集
- 型番正規表現の追加: SE が変更レビュー必要
- LLM プロンプト変更: SE + センター長の合議

### 11-4. メンテナンス
- 運用主体: CS 部 運用担当 2 名
- 監視: Datadog（レイテンシ・エラー率・コスト）
- 月次改善会: ベテランオペ 2 名 + SE 1 名で「ドラフトが微妙だったケース」をレビュー

## 12. 既知のリスクと対処

| # | リスク | 重大度 | 対処 |
|---|---|---|---|
| 1 | 誤った納期 / 在庫情報を顧客に送信 | 致命 | DB 引き当て結果をコード側で必ずチェック、LLM が数字を捏造する余地を残さない（プロンプトで明示禁止 + 値プレースホルダ方式）|
| 2 | クレーム見逃しで顧客大激怒 | 高 | キーワード辞書 + LLM 二重判定。再現率優先（過検知は SV が捌く）|
| 3 | 顧客個人情報の外部漏えい | 高 | LLM 送信前マスキング層（氏名 / 電話 / 住所）。マスキング前後ログを月次照合 |
| 4 | コスト爆発 | 中 | recursion_limit、1 件最大トークン上限、月予算の 80% でアラート |
| 5 | LLM API ダウン | 中 | テンプレ定型文 + 「AI 生成失敗、手動対応してください」フラグ |
| 6 | DB ダウン | 中 | 「在庫情報取得失敗。担当より折り返しご連絡」テンプレ + 内部通知 |
| 7 | 抽出型番が DB にない | 中 | 「在庫情報を確認中」テンプレ + 内部通知（型番マスタへの追加候補リスト化）|

## 13. 採用する設計パターン

- [x] **Tool Use**（外部 DB / API 連携が中心）
- [x] **Reflection**（クレーム検出・回答の自己チェック）
- [ ] Planning（v1 では不要、シンプルなパイプラインで十分）
- [ ] Multi-Agent（v1 不要。v2 以降で Planner+Reviewer+Synthesizer 化を検討）

選定理由:
- 業務はおおむね「分類 → 抽出 → DB 引き当て → 起草」の決定論的順序で固定できるので、純エージェント（自律ループ）よりも **ハイブリッドなワークフロー型（LangGraph StateGraph）** が適切
- LLM の自由度は最小化し、DB 引き当ては Tool Use として固定。LLM は「カテゴリ判定」「逆質問の選択」「下書き起草」「クレーム匂い検出（二重判定）」に限定する
- 「下書き」を出した後に **自己レビュー（Reflection）** で「数字をでっち上げてないか」「テンプレ語が入っているか」を確認する 1 ステップを挟む

> 詳細は `/agent-decompose` で `design.md` に展開する。

## 14. 採用する技術スタック

- フレームワーク: **LangGraph**（StateGraph） + **LangChain Anthropic**
- LLM プロバイダ: **Anthropic**
  - 主要モデル: **Claude Sonnet 4.6**（カテゴリ判定 / 起草 / Reflection）
  - 軽量モデル: **Claude Haiku 4.5**（型番抽出補助 / 定型ケース起草）
- ベクトル DB: **不使用**（v1）。FAQ 検索は v2 以降の検討
- 永続化: **SQLite**（ケース ID と AI ドラフトの紐付け、重複排除用）
- 配信: Salesforce REST API + Slack（クレーム時通知）
- 設定駆動: **YAML**（カテゴリキーワード / 型番正規表現 / テンプレ文言）
