## 「やらない」と決めたユースケース

リソース有限のため、以下は本パックの対象外:

| ユースケース | やらない理由 |
|---|---|
| AI が顧客に自動送信 | ブランド毀損リスク。必ず人間レビュー前提 |
| 与信判断・支払い遅延の督促文生成 | 法務・与信判断の責任が重く、AI 生成は不適切 |
| クレームの最終回答（謝罪文の最終版）の生成 | SV 判断必須。AI は「お詫び一言 + フラグ」までに限定 |
| 価格交渉の代行 | 値引き約束は AI 禁止（spec §6 ルール） |
| 競合製品との比較推奨 | 比較は事実情報のみ、推奨は営業判断 |

## 着手順（v1）

LayerX 指針: **エンジニア主導で技術突破しやすいトピックから**。

```
Step 1. ケース 1, 3（在庫照会 / 配送状況）の動くプロトタイプ
  → DB モック + 型番抽出（regex） + テンプレ起草。技術突破が容易
Step 2. ケース 4（クレーム匂い検出）を Reflection で追加
  → キーワード辞書 + LLM 二重判定
Step 3. ケース 5, 6（廃番後継 / 急ぎ逆質問）で「裏の意図」ロジック追加
Step 4. ケース 7（技術質問の逆質問）と ケース 9（経理引き継ぎ）で文体ルール拡張
Step 5. ケース 10（必須テンプレ語含有チェック）を Reflection 内で実装
```

ここまで動けば B ランクは YAML 追加で大半が捌けるはず（YAML 駆動の威力）。

## カタログの維持

- 月次で完了ユースケースの実績値を更新（予想 6 分→4 分 / 実測どうだったか）
- 新規ユースケースを追加（運用ログから発見）
- 優先度の見直し（運用しながら判明した事情を反映）
- やらないリストにも追加（再検討が来た時のため）

---

## カタログ → 設計への接続

- v1 が動いたら、**ケース 23（英語）／ ケース 25（FAQ）** が「テーマ追加だけで動く」可能性あり
- ケース 21〜22（OCR / Excel 解析）は v2 で別ノード追加が必要 → 設計パターンの拡張が必要
- ケース 26〜30（分析・改善系）は別エージェントとして派生（共通基盤を活用）
