> 作成日: 2026-05-09 / 作成者: agent-discover (Mode A)
> 入力: `data/_initial_brief.md`, `data/hearing_notes.md`

---

## 1. 目的

中堅企業（300 名規模）の経理部に毎月届く約 1,500〜2,000 件の経費 / 請求申請を **ルール自動チェック + LLM グレーゾーン判定** でレビューし、`auto_approve / needs_fix / needs_review / reject` の 4 段階に振り分ける。経理の手数を月 125h → 60h に半減し、差し戻し率 18% → 10% を狙う。

## 4. 編集者ペルソナ

```
あなたは中堅 SaaS 企業のベテラン経理担当（勤続 9 年）。月 600 件の経費申請を 1 人で
捌く現場リーダー。金額の中途半端さを嗅ぎ取り、店名と勘定科目の不整合を見抜き、
重複申請とグレーゾーンを早期に察知するのが得意。申請者には敬体で具体的に
フィードバックし、再申請しやすい修正例を必ず添える。判断に迷うグレーは
人手レビューに回す。最終承認は経理マネージャと監査の領分で、あなたは「自動 OK」
「軽微修正でいける」「人手判断必要」を仕分ける役。
```

---

## 5. 編集者の視点（判断ルール、ベテラン経理 田島の暗黙知）

1. **金額の丸め方**: ぴったり 10,000 / 5,000 / 50,000 円のような丸数字は申請者操作の可能性、領収書本文の確認を要する
2. **店名と勘定科目の整合性**: 高級店 × 会議費、居酒屋 × 接待 + 相手社内 等の不整合はグレーフラグ
3. **重複検出**: 同一申請者・同一店名・同一金額の過去 90 日以内の重複は高リスク
4. **接待要件**: 「相手先空欄 / 人数空欄 / 社員のみ参加」は社内飲み会混入の可能性
5. **出張規程**: 一人出張で接待単価上限超は規程違反
6. **精算期限**: 領収書日付が 60 日以上前は原則差し戻し
7. **承認権限**: 金額上限超は権限者承認（部長・本部長）の有無を確認
8. **科目変更頻度**: 同申請者が同類の経費を毎月別科目で申請している場合は要警戒

## 6. 執筆・出力ルール

| カテゴリ | ルール |
|---|---|
| 必須含有（フィードバック文言）| 「以下の点をご確認ください」/ 「修正後の再申請をお願いいたします」/ 具体的な指示語 |
| 絶対に書かない | 「絶対に承認できません」「ルール違反です」（断定的・否定的すぎる）/ 内部用語（「経理判断」「上の判断」）/ 個人攻撃 |
| トーン | 敬体・建設的。「ルールに沿うように修正案 X / Y / Z をご提示します」 |
| 数値 | 領収書 / 申請の値そのまま。LLM が金額を改変しない |
| 不確実性 | 「規程に照らして要確認のため、経理担当より連絡します」（needs_review 時）|
| 引き継ぎ | needs_review 時は内部メモに「どのルールでどの方向に倒すか迷ったか」を 100 文字以内で記録 |

## 10. 制約

### 10-1. コスト
- 1 件: $0.03 以下（cs_triage より厳しめ）
- 月: $1,000（1,500〜2,000 件 × $0.03 ≈ $45〜60、Reflection 込みで余裕）
- 軽量モード: あり（Haiku のみ、auto_approve 候補の単純ケース）

### 10-2. 速度
- webhook 受信 → 書き戻し: 15 秒以内（楽楽精算 webhook タイムアウトに合わせる）
- バーストはキューイング許容

### 10-3. セキュリティ
- 機密度: **個人情報含む**（申請者氏名・社員 ID）
- LLM API 送信: OK。氏名・社員 ID マスキング、領収書画像本体は送らず OCR 抜粋のみ
- ログに残す: 申請 ID、判定、参照ルール、フィードバック、信頼度
- ログに残さない: 氏名生データ、社員 ID 生データ、口座情報
- 監査: 月次でランダム 100 件をマスキング前後照合

### 10-4. 依存
- 楽楽精算 / freee / 社内 ID（Okta） / Slack / Datadog

## 11. 運用条件

### 11-1. トリガー
- イベント駆動: 楽楽精算 webhook（申請者の申請ボタン押下 or 上長承認完了）
- バッチ: 月末月初バーストはキュー経由
- リトライ: 3 回（指数バックオフ）

### 11-2. レビュー工程
- needs_review: 経理担当が楽楽精算上で最終判断
- 自動承認時: 月次抜き取り 100 件を SV がスポットレビュー
- 監査向け: rule_trace JSON を四半期 CSV エクスポート

### 11-3. 変更管理
- expense_policy.yaml: 経理マネージャが編集 + SE レビュー
- LLM プロンプト変更: SE + 経理マネージャ合議

## 12. 既知のリスクと対処

| # | リスク | 重大度 | 対処 |
|---|---|---|---|
| 1 | 不正経費を auto_approve | 致命 | コードでルール検査を必ず通し、ハードチェック（金額・領収書・期限）を LLM 判定の前段に。LLM は OK 判定の **追加チェック層** として位置づけ |
| 2 | 重複申請見逃し | 高 | 過去 90 日 DB 検索を必ず通す、検出しきい値は YAML で調整可 |
| 3 | 個人情報漏えい | 高 | LLM 送信前マスキング、月次 PII 監査 |
| 4 | フィードバック文言の抽象性 | 中 | LLM プロンプトに「具体的な修正例 1〜2 個提示」を強制、reflect でチェック |
| 5 | コスト爆発（バースト時）| 中 | recursion_limit、月予算 80% で軽量モード強制、100% で停止 |
| 6 | 楽楽精算 API ダウン | 中 | webhook 受信時キュー退避、復旧後リトライ |
