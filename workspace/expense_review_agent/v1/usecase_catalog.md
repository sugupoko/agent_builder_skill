## 「やらない」と決めたユースケース

| ユースケース | やらない理由 |
|---|---|
| AI による不正の最終認定 | 法務 / 監査 / 役員判断必須、AI は「疑わしい」フラグまで |
| 申請者への自動却下メール送信（人手レビューなし）| 心理的ダメージ大、必ず経理経由で連絡 |
| 銀行口座情報の修正提案 | 別系統の業務 |
| 給与計算との完全自動連携 | 計算ミスの責任が AI に転嫁できない |
| カード明細からの自動経費起票 | 申請者の意図確認なしには不可 |

## 着手順（v1）

LayerX 指針: **エンジニア主導で技術突破しやすいトピックから**。

```
Step 1. ケース 1 (出張交通費 auto_approve) — 単純ルール検査で OK、技術突破容易
Step 2. ケース 9 (科目選択ミス needs_fix) — フィードバック文言生成のテンプレ化
Step 3. ケース 2, 3 (接待要件) — Reflection でグレー判定の精度確認
Step 4. ケース 4 (重複検出) — 過去申請 DB 引き当て、Tool Use の典型
Step 5. ケース 7 (一人出張規程違反 reject) — ルール組み合わせ判定
Step 6. ケース 10 (監査ログ生成) — rule_trace の整備で監査要件を満たす
```

ここまで動けば B ランクの 11〜20 は YAML 追加で大半が捌ける。

---

## カタログ → エージェント設計への接続

- v1 (本実装): 4 判定（auto_approve / needs_fix / needs_review / reject）の振り分けに集中
- v2: 過去事例 RAG（ケース 22）、規程違反常習者検出（ケース 23）
- v3: 多言語 / 法人カード突合 / インボイス制度

各カテゴリ（出張・接待・会議・消耗品・通信・研修・雑費）は YAML の `expense_policy` で完結する設計のため、v1 後の追加カテゴリは config 編集のみで対応可能。
