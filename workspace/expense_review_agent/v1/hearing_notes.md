## 1. 業務理解

### 1-1. 目的（中村）
- 月 1,500〜2,000 件の経費申請を 経理 4 名で捌く現状の **属人化（田島依存 600 件）解消** が主目的
- 差し戻し率 18% → 10% を狙う（差し戻しの 7 割は単純ミス）
- 経理の手数 月 125h → 60h（半減）

### 1-2. 業務ステップ（6 ステップ）
```
1. 申請者が楽楽精算で申請（領収書添付、金額・日付・科目・相手先記入）
2. 上長承認（金額大、出張、接待）
3. 経理が受領、ルールチェック（自動 + 目視）
4. 不備があれば差し戻し、軽微なら経理補正、判断要は田島へエスカレ
5. OK なら freee 連携、振込予定リスト
6. 監査ログ生成、月末締めデータ集約
```

→ **エージェント化したい範囲**: 3 のうち (a) ルール自動チェック (b) フィードバック文言生成 (c) 過去申請との重複検出 (d) 要人手判断の振り分け

### 1-3. 「人の判断スキル」（田島の暗黙知 → 言語化）

**3 つのスキル**:

1. **金額の中途半端さの読み**: 1 万円ぴったり vs 9,800 円。丸数字は申請者操作の可能性
2. **店名と勘定科目の整合性**: スターバックスで会議費は OK、銀座フレンチで会議費は怪しむ
3. **重複・グレーの嗅覚**: 同店名・同金額の過去 90 日重複、相手先空欄の接待、一人出張で接待単価超

### 1-4. editorial.persona の素材（重要）

```
1. 金額の丸め方を見る: ぴったり 10,000 / 5,000 / 50,000 円は申請者操作の可能性、領収書本文確認
2. 店名と勘定科目の整合性: 高級店 × 会議費、居酒屋 × 接待 + 相手社内 等の不整合を疑う
3. 同一申請者・同一店名・同一金額の過去 90 日以内の重複を必ず確認
4. 接待で「相手先」「人数」が空欄 / 社員のみは要警戒（社内飲み会の混入）
5. 出張で「一人出張 + 接待単価上限超」は規程違反の可能性
6. 領収書日付が 60 日以上前は精算期限超過、原則差し戻し
7. 金額上限超は権限者承認の有無を確認、なければ差し戻し
8. 科目変更（消耗品 → 備品など）が頻繁な申請者は別途レビュー
```

→ そのまま `spec.md §5 編集者の視点` に入る。

### 1-5. 申請者ペインポイント（川口）
- 差し戻し理由の定型化で「何が悪いか不明」
- 出張規程と接待規程の上限が違う点が分かりにくい
- → **フィードバック文言は具体的修正例を必ず添える**

### 1-6. 監査の観点（佐々木）
- 重複申請の見逃し、規程違反常習、監査ログ欠損
- **判定理由を必ずログに残す**（rule_trace JSON、四半期 CSV エクスポート）
- 抜き取り監査用にランダムサンプリング 月 100 件継続

## 2. 入出力

### 2-1. 入力
- 種類: 楽楽精算 webhook の構造化 JSON
- 形式: 申請ID / 申請者 / 申請日 / 領収書日付 / 金額 / 科目 / 摘要 / 相手先 / 添付 URL + OCR テキスト
- サイズ: 1〜5 KB / 件
- 流入: 平均 1 件/数分、ピーク 30 件/時間（月末月初）
- ソース: 楽楽精算 API

### 2-2. 出力
- 種類: 判定 JSON + 申請者向けフィードバック + 内部メモ + 監査用 rule_trace
- 配信先: (1) 楽楽精算書き戻し / (2) Slack `#expense-feedback` 申請者通知 / (3) Slack `#expense-audit` 経理通知
- 量: 申請者向け 100〜300 字、判定 JSON 8〜15 フィールド
- 受け取り手: 申請者（needs_fix / reject 時）/ 経理（needs_review）

## 3. 成功基準と評価

### 3-1. 「うまくいった」の定義
1. **経理手数 50% 削減**: 月 125h → 60h
2. **差し戻し率 18% → 10% 以下**
3. **重複申請検出件数 +50%**（監査品質）
4. 絶対条件: **判定理由が必ずログに残る**（佐々木）

### 3-2. 評価データ計画
合成 12 件をシード:
- auto_approve 3 / needs_fix 4 / needs_review 3 / reject 2

### 3-3. 評価軸
- **コードベース**: 判定一致率、ルール参照漏れ、フィードバック必須語含有
- **LLM as a judge**: 具体性 / トーン / 申請者が動きやすいか
- **人間評価**: 月次で田島 + 佐々木が抜き取り 30 件評価

## 4. 制約

### 4-1. コスト
- 1 件: $0.03 以下
- 月: $1,000 上限（月 1,500〜2,000 件想定）
- 軽量モード: あり（Haiku のみ、auto_approve 候補）

### 4-2. 速度
- webhook 受信 → 書き戻し: 15 秒以内（楽楽精算のタイムアウト合わせ）
- バーストはキューイング許容

### 4-3. セキュリティ
- 機密度: 個人情報含む（申請者氏名・社員 ID）
- LLM API: OK、氏名・社員 ID マスキング
- 領収書画像本体は LLM に送らず OCR 抜粋のみ
- ログに残す: 申請 ID、判定、参照ルール、フィードバック、信頼度
- ログに残さない: 氏名生データ、社員 ID 生データ、口座情報
- 監査: 月次 PII 監査ランダム 100 件

### 4-4. 依存
- 楽楽精算 / freee / Okta（社内 ID）/ Slack / Datadog
