## 4. 編集者ペルソナ（最重要）

ベテラン経理「田島」の暗黙知を 8 項目に言語化:

1. 金額の丸め方を見る（10,000 / 5,000 円ぴったりは要確認）
2. 店名と勘定科目の整合性（高級店 × 会議費は不整合）
3. 同一申請者・同一店名・同一金額の過去 90 日重複検出
4. 接待で「相手先 / 人数」空欄 / 社員のみは要警戒
5. 一人出張 + 接待単価超は規程違反
6. 領収書日付 60 日以上前は精算期限超過
7. 金額上限超は権限承認の有無確認
8. 科目変更頻度の高い申請者は別途レビュー

→ `spec.md §4 / §5` に転記。

## 5. 評価データ計画

合成 12 件をシード:

| 判定 | 件数 | カバレッジ |
|---|---|---|
| auto_approve | 3 | 出張交通費 / 通常会議費 / 規程内消耗品 |
| needs_fix | 4 | 接待相手先空欄 / 領収書なし / 科目選択ミス / 上長承認漏れ |
| needs_review | 3 | 接待単価超グレー / 重複疑い / 一人出張規程違反疑い |
| reject | 2 | 60 日超期限切れ / 一人出張で接待単価明確違反 |

評価軸（3 軸）:
- **コードベース**: 判定一致率、ルール参照漏れ、フィードバック必須語含有
- **LLM as a judge**: 具体性 / トーン / 申請者が動きやすいか
- **人間評価**: 月次で田島 + 佐々木が抜き取り 30 件

## 8. 設計上の所見（次スキルへのメモ）

### Decompose 担当への申し送り

- **採用パターン**: Tool Use（policy / 社員マスタ / 過去申請 DB）+ Reflection（フィードバック具体性チェック）
- **推奨ノード分割**（仮）:
  ```
  1. preprocess（PII マスク、領収書 OCR 抜粋整理）
  2. extract（金額・日付・店名・相手先・人数を構造化抽出）
  3. validate_rules（コードでハードチェック: 金額上限 / 期限 / 必須項目 / 重複）
  4. classify_gray（LLM: グレーゾーン（接待単価超など）の判定）
  5. lookup_history（過去 90 日重複検出、Tool Use）
  6. draft_decision（LLM: 4 判定 + フィードバック文言生成）
  7. reflect（LLM: 具体性・ルール参照漏れ・トーン）
  8. assemble（rule_trace JSON 含む最終出力）
  ```
- **モデル使い分け**:
  - Sonnet 4.6: classify_gray / draft_decision / reflect
  - Haiku 4.5: extract（OCR 補完） / 単純 auto_approve 用 draft
- **YAML 駆動範囲**: expense_policy（科目・上限金額・規程ルール）、フィードバックテンプレ

### Prototype 担当への申し送り

- DB は **モック（YAML / JSON フィクスチャ）** で開始
- PII マスキング: 申請者氏名・社員 ID
- regex は YAML から（テストドリフト防止）
- TypedDict 全フィールド宣言
- Sonnet/Haiku 切替は config 経由

### Eval 担当への申し送り

- 評価軸 9 項目（spec §9）のうち最重要: **#2 auto_approve false positive 0 件（絶対）**、**#4 重複検出再現率 95%**、**#1 判定一致率 85%**
- LLM Judge プロンプトは「ベテラン経理 田島」視点で 5 次元採点
- フィードバック文言の具体性（「修正例 X / Y / Z」が含まれているか）を Judge の info_completeness で見る

## 10. cs_triage_agent との関係

`workspace/cs_triage_agent/` がアシスタント系 + 抽出系 + レビュー系の複合サンプルだったのに対し、本 `expense_review_agent` は **純レビュー系** にフォーカス。スキルパックの 4 種別カバレッジを補完する位置付け。

| 観点 | cs_triage_agent | expense_review_agent |
|---|---|---|
| エージェント種別 | アシスタント + 抽出 + レビュー | **レビュー（純）**|
| 入出力 | メール → 回答ドラフト | JSON 申請 → 判定 + フィードバック |
| LLM 役割 | 起草 / 分類 / 自己レビュー | グレーゾーン判定 / フィードバック起草 / 自己レビュー |
| 決定論コード比率 | 中（DB 引き当ては決定論）| **高**（ルール検査が決定論の主役）|
| 致命リスク | クレーム見逃し | 不正 auto_approve |
| 主要 KPI | 応対時間 30% 短縮 | 経理手数 50% 削減 |
