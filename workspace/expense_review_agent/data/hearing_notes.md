## A. 業務の理解

### A-1. 業務の目的（中村 → 全員共通）

> **中村**: 「うちは 300 名規模で、月 1,500〜2,000 件の経費申請が来ます。経理 4 名で捌いてますが、月末月初の集中で残業が常態化してます。ベテランの田島が 600 件 1 人で捌いてて、彼女が休むと精算サイクルが回らなくなる。」
>
> 「KPI は月次決算締めまでの日数と差し戻し率。差し戻し率は今 18% で、これを 10% 切りたい。差し戻しの 7 割は『金額上限超過』『領収書なし』『科目選択ミス』『相手先空欄』のような単純ミス。残り 3 割が『規程違反かどうかグレー』『前月に類似申請があった疑い』のような判断系。」
>
> 「自動レビューで単純ミスを申請者に直接フィードバックして再申請してもらえれば、経理の手数を半分にできるはず。」

- **やめたら困る人**: 申請者（精算が遅れて立替金回収が遅れる）、経理（属人化解消）、監査（ログ品質）
- **手作業時間**: 田島 600 件 / 月 × 5 分 = 50h、他 3 名計 900 件 × 5 分 = 75h、合計 **約 125h/月**

### A-2. 業務ステップ分解

```
1. 申請者が楽楽精算で申請（領収書添付、金額・日付・科目・相手先記入）
2. 上長承認（金額大きいもの、出張、接待）
3. 経理が受領、システム上でルールチェック（自動 + 目視）
4. 不備があれば差し戻し、軽微なら経理側で補正、判断要は田島へエスカレ
5. OK なら freee へ連携、振込予定リストへ
6. 監査ログ生成、月末締めデータに集約
```

→ **エージェント化したい範囲**: 3 のうち **(a) ルール自動チェック (b) フィードバック文言生成 (c) 過去申請との重複検出 (d) 要人手判断の振り分け**。最終承認 (5) は必ず人間。

### A-3. 「人の判断スキル」の所在（田島から引き出す）

> **田島**: 「経験が効くのは 3 つ。**金額の中途半端さの読み**、**店名 / 科目の整合性チェック**、**重複・グレーの嗅覚**。」
>
> 「金額の中途半端さ — 1 万円ぴったりの接待 vs 9,800 円の接待。後者は『税抜き 9,800 円じゃないと申請したくない』申請者の操作の可能性。明らかに丸めた数字は信用しない。」
>
> 「店名と科目 — スターバックスで会議費は OK だが、銀座の高級フレンチを会議費にしてくると怪しむ。店の業態と金額帯で『これは接待だろう』と気づく。」
>
> 「重複・グレー — 同じ店名・同じ金額が前月にもあったら高確率で重複申請。あと、相手先空欄の接待は『社内飲み会を接待でごまかしてる』可能性。一人出張で接待単価上限超えも要警戒。」

#### 言語化された判断スキル（editorial.persona の素材）

```
1. 金額の丸め方を見る: ぴったり 10,000 / 5,000 円のような丸数字は申請者操作の可能性、領収書本文確認
2. 店名と勘定科目の整合性: 高級店 × 会議費、居酒屋 × 接待 + 相手社内 等の不整合を疑う
3. 同一申請者・同一店名・同一金額の過去 90 日以内の重複を必ず確認
4. 接待で「相手先」「人数」が空欄 / 社員のみは要警戒（社内飲み会の混入）
5. 出張で「一人出張 + 接待単価上限超」は規程違反の可能性
6. 領収書日付が 60 日以上前は精算期限超過、原則差し戻し
7. 金額上限超は権限者承認の有無を確認、なければ差し戻し
8. 科目変更（消耗品 → 備品など）が頻繁な申請者は別途レビュー
```

### A-4. 申請者ペインポイント（川口）

> **川口**: 「差し戻しが来ると 1 件あたり 30 分かかります。理由が『領収書再添付』だけだと、何が悪いのか分からない。『日付が 7/15 になってますが、申請月は 5 月です』みたいに具体的に書いてくれると即対応できる。」
>
> 「あと、出張規程と接待規程で上限が違うのが分かりにくい。『この申請は接待規程の単価 5,000 円を超えてます。理由を追記してください』みたいに具体的に教えてくれるといい。」

### A-5. 内部監査の観点（佐々木）

> **佐々木**: 「監査で困るのは 3 つ。**重複申請の見逃し**、**規程違反の常習**、**監査ログの欠損**。重複申請は申請者と店名と金額のキー突合をしないと見つからない。常習は『この申請者は 3 ヶ月連続で接待単価超』のような時系列パターン。ログは『なぜ自動承認になったか』の根拠が残ってないと指摘する。」
>
> 「自動レビュー入れるなら、**判定理由（どのルールに照らしたか）を必ず保存**してほしい。あと、抜き取り監査用にランダムサンプリングは月 100 件継続してほしい。」

## B. 入力と出力

### B-1. 入力

| 項目 | 値 |
|---|---|
| 種類 | 構造化フォーム（楽楽精算の API 出力 JSON） |
| 形式 | JSON: 申請ID / 申請者 / 申請日 / 領収書日付 / 金額 / 科目 / 摘要 / 相手先 / 添付ファイル URL |

## C. 成功基準と評価

### C-1. 「うまくいった」の定義（中村 + 佐々木）

> **中村**: 「現場の感覚的な KPI は 3 つです。」
> 1. **経理の手数 50% 削減**: 月 125h → 60h
> 2. **差し戻し率 18% → 10% 以下**: 申請者の再申請負担も減る
> 3. **重複申請の検出件数 +50%**: 監査品質向上

> **佐々木**: 「私の絶対条件は 1 つ。**判定理由が必ずログに残ること**。何のルールに照らして自動承認したかが追えれば、監査時に問題ない。」

### C-2. 評価データ

合成データで開始。判定別に:

| 判定 | 件数 | 良い出力の特徴 | 悪い出力の特徴 |
|---|---|---|---|
| auto_approve | 3 | 全ルール OK の素直な申請を即承認、申請者通知不要 | 些細な不備で auto_approve（監査リスク）|
| needs_fix | 4 | 軽微な誤りを具体的に指摘 + 修正例提示、申請者が再申請しやすい | 「再添付してください」だけ（具体性不足）|
| needs_review | 3 | グレーゾーンを経理 / 田島へ「なぜ要レビューか」を添えて引き継ぎ | 自動判断してしまう or 経理に丸投げ |
| reject | 2 | 規程違反を断固却下、根拠ルール明示 | 拒否理由が曖昧 |

合計 **12 件**でシード。

### C-3. 評価方法

- **コードベース**: 判定一致率（期待 vs 実測）、ルール参照漏れ、フィードバック文言の必須語含有、文字数レンジ
- **LLM as a judge**: フィードバック文言の具体性 / トーン適切性 / 申請者が動きやすいか
- **人間評価**: 月次で田島 + 佐々木が抜き取り 30 件を評価

## D. 制約と非機能要件

### D-1. コスト
- 1 件あたり予算: **$0.03 以下**（cs_triage より厳しめ、件数多いため）
- 月間上限: **$1,000**（月 1,500〜2,000 件 × $0.03 = $45〜60 だが、Reflection 込みで $1,000 までは許容）
- 軽量モード: あり（Haiku のみ、auto_approve 候補の単純ケース）

### D-2. 速度
- 受信から判定書き戻しまで: **15 秒以内**（楽楽精算側の webhook タイムアウトに合わせる）
- バッチ可（月末バーストはキューイング）

### D-3. セキュリティ
- データ機密度: **個人情報含む**（申請者氏名・社員 ID・銀行口座関連は別系統）
- 外部 LLM API 送信: **OK**（NDA 想定）。氏名・社員 ID はマスキング、領収書画像本体は送らない（OCR 抜粋のみ）
- ログに残す: 申請 ID（マスク後）、判定、参照ルール、フィードバック文言、信頼度
- ログに残さない: 申請者氏名生データ、社員 ID 生データ、口座情報
- 監査: 月次でランダム 100 件をマスキング前後照合

### D-4. 依存関係
- 楽楽精算 API（必須、申請データの起点）
- freee API（連携先、判定結果書き戻し）
- 社内 ID 管理（Active Directory or Okta）
- Slack（フィードバック・監査通知）

## E. 運用と継続性

### E-1. トリガー
- イベント駆動: 楽楽精算 webhook（申請者の「申請ボタン押下」or「上長承認完了」イベント）
- バッチ: 月末月初のバースト時はキュー経由で処理

### E-2. レビュー工程
- **必須**: needs_review 判定時は経理担当（田島など）が SF / 楽楽精算上で最終判断
- 自動承認時も監査向けに月次抜き取り 100 件を SV がスポットレビュー
- 監査向け: 全判定の rule_trace JSON を CSV で四半期エクスポート

### E-3. 変更管理
- 規程・上限金額・科目定義: YAML 編集（経理マネージャ 中村が直接編集）
- 重複検出ロジック・LLM プロンプト: SE 藤本 + 中村の合議

### E-4. メンテナンス
- 運用主体: 経理部 + SE 1 名兼任
- 監視: Datadog（レイテンシ・コスト・誤判定率）
- 月次改善会: 田島 + 佐々木 + SE 藤本で「微妙な判定」を再レビュー

## F. リスクと失敗モード

### F-1. 重大度順

| # | リスク | 重大度 | 顕在化シナリオ |
|---|---|---|---|
| 1 | 不正経費を auto_approve | 致命 | ルール検査漏れ + LLM が「OK」判定 → 監査で発覚 |
| 2 | 重複申請の見逃し | 高 | 過去 90 日の検索範囲漏れ |
| 3 | 個人情報漏洩 | 高 | マスキング漏れで LLM API に氏名・社員 ID 送信 |
| 4 | 申請者への差し戻し理由が不明瞭で再申請ループ | 中 | LLM のフィードバック文言が抽象的 |
| 5 | コスト爆発 | 中 | Reflection 過剰、月末月初のバースト |
| 6 | 楽楽精算 API ダウン | 中 | webhook 受信不能、judging 結果書き戻し失敗 |

### F-2. フォールバック

- LLM ダウン: 「ルール自動チェックのみ」（needs_review 寄りに倒す、安全側）+ 「AI 自動レビュー一時停止中」フラグ
- DB 不通（社員マスタ・過去申請）: needs_human_review に倒す
- 楽楽精算 API ダウン: webhook 受信時にキューに退避、復旧後リトライ
