# Agent Mindset

AIエージェントを設計・実装するときに繰り返し効く7つの思考回路と、設計レビュー前のチェックリスト。

---

## 7つの思考回路

### 1. LLM には判断させすぎない

**症状**: 何でも LLM に聞いて全工程を生成させる → コスト増、再現性なし、デバッグ困難。

**対処**: 工程を「**決定論コードで済む処理**」と「**LLM 判断が必要な処理**」に分ける。

| 決定論コードでやる | LLM でやる |
|---|---|
| データ収集（API/RSS/DB） | 要約・編集 |
| フィルタ・並び替え・集計 | 分類・関連性判断 |
| 重複排除・正規化 | 自然言語の解釈 |
| ファイル出力・配信 | 文章生成 |

LLM は最後の「人間が読む文章を作る」段階に集中させる。

---

### 2. ハイブリッド構成で始める

純粋な「自律エージェント」ではなく、**基本はワークフロー型で、一部だけエージェント化**するのが実務の主流。

```
fetch → process → summarize → enrich → review → send
                       ↑                  ↑
                  LLM 1回             ReAct エージェント
```

- 全体の流れは決定論的にコード制御（LangGraph の StateGraph）
- 「背景調査」「補強検索」など探索が必要な箇所だけ ReAct
- これでコスト・予測可能性・デバッグのバランスが取れる

参考: [reference/workflow_vs_agent.md](reference/workflow_vs_agent.md)

---

### 3. 評価データから始める（Eval-driven）

**症状**: 出来てから「これでいいか分からない」状態になる。

**対処**:
- ヒアリング段階で「**良い出力 / 悪い出力**」のサンプルを 10〜20 件もらう
- 評価関数を**最初に作る**（精度・カバレッジ・コスト・所要時間）
- 評価データは少数で OK。LayerX も「代表的な十数個」から始めると言っている

参考: [reference/eval_driven_dev.md](reference/eval_driven_dev.md)

---

### 4. 編集者の声 / ペルソナをプロンプトに注入

**症状**: LLM出力が汎用的・空虚（「業界が活況」「期待が高まる」）。

**対処**: ユーザーの判断スキル・視点・執筆ルールを言語化して**プロンプトに注入**する。

```yaml
editorial:
  persona: |
    20年の○○業界経験を持つ編集者。△△ に詳しい。
  perspective: |
    - 「で、ウチはどう動くか」を常に問う実務志向
    - 広報的成功談より現場のつまずきに着目
  rules: |
    - 汎用的な感想は禁止
    - 数字・固有名詞を必ず含める
    - 同案件の重複ソースは1件にマージ
```

これだけで出力の質が劇的に変わる。

参考: [reference/editorial_persona.md](reference/editorial_persona.md)

---

### 5. リランキングで脱線を防ぐ

**症状**: ReAct が深掘りで的外れな記事を選ぶ、サマリが要点を外す。

**対処**: 取得した情報を「**テーマ関連性スコア**」で並び替えてから LLM に渡す。

```python
def score(item, theme_keywords):
    title = item.title.lower()
    snippet = item.snippet.lower()
    return sum(2 if k in title else (1 if k in snippet else 0) for k in theme_keywords)

items = sorted(items, key=lambda it: score(it, kws), reverse=True)
```

サマリの入力（先頭60件）も、深掘り対象（スコア最上位）も、これだけでテーマ忠実性が跳ね上がる。

参考: [reference/reranking_guide.md](reference/reranking_guide.md)

---

### 6. ツールは最小権限・キャッシュ・ループ制御

**症状**: ReAct が同じ URL を何度も叩く、ツール呼び出しが暴走、権限事故。

**対処**:
- **キャッシュ**: 同一 URL/クエリの2回叩きをプロセス内で防止
- **最小権限**: ツールが必要な範囲のみアクセス可能に
- **ループ制御**: `recursion_limit` を設定 + プロンプトで「最大N回」と明示
- **エラー時の代替**: ツール失敗時は他ツールへフォールバック

参考: [reference/tools_pattern.md](reference/tools_pattern.md)

---

### 7. YAML で運用、コードを触らない

**症状**: テーマ追加・キーワード調整のたびに Python を編集 → エンジニア依存。

**対処**: 設定はすべて YAML に切り出す。

```yaml
theme: ...
sources: [...]
editorial:
  persona: ...
sections:
  weekly_todos: true
  vendor_watchlist: true
exclude_keywords: [...]
trusted_publishers:
  blocklist: [...]
```

業務担当者が YAML を直接編集できるレベルまで持っていく。コード変更は本当に必要なときだけ。

## 設計レビュー前チェックリスト

エージェントを実装する前、または LLM に「動くものを作って」と頼む前に、以下を全て答えられるか確認する。

### 仕様の明確性

- [ ] **入力**: 何が来る？（自然言語？構造化？ファイル？）
- [ ] **出力**: 何を返す？（テキスト？Markdown？JSON？ファイル？）
- [ ] **トリガー**: いつ実行？（手動？cron？イベント駆動？）
- [ ] **頻度**: 1日何回？1週間何回？
- [ ] **読み手**: 誰が読む？（自分？上司？取引先？）

### 評価可能性

- [ ] **良い出力の例**を3つ以上書ける？
- [ ] **悪い出力の例**を3つ以上書ける？
- [ ] **評価指標**を3つ定められる？（精度・コスト・所要時間・カバレッジ等）
- [ ] **評価データセット**を10件作れる？

### LLM/コード境界

- [ ] **決定論コードで済む処理**を3つ列挙できる？
- [ ] **LLM が必要な処理**を1〜2つに絞れる？
- [ ] LLM が落ちたとき、システム全体は動く（fallback）？

### コスト

- [ ] **1回の実行コスト**の見積もり（トークン × 単価）が出る？
- [ ] **月間予算**を上限として設定している？
- [ ] **軽量モード**（深掘りスキップ等）を用意している？

### 失敗の想定

- [ ] **ツール失敗時**の挙動は？（リトライ？スキップ？失敗を残す？）
- [ ] **LLM の出力が想定外**だったときの対処は？
- [ ] **配信失敗時**のリカバリは？

### 運用

- [ ] **ログ**: 1実行1ファイルで残す
- [ ] **コスト計測**: usage_metadata から推定
- [ ] **バージョン管理**: spec.md がそのバージョンの「正」
- [ ] **配信先の変更**を YAML で済ませられる？

## TypedDict / LangGraph での落とし穴

LangGraph の `StateGraph(TypedDict)` は **TypedDict で宣言されたキーしか保持しない**。
未宣言のキーは silent drop される。`state.get("foo", default)` のデフォルト値が常に使われる罠。

```python
class WorkflowState(TypedDict, total=False):
    cfg: dict
    items: list
    # ↓ これを書き忘れると state["n_topics"] が消えて get() のデフォルトに引きずられる
    n_topics: int
    enable_thinking: bool
```

**ルール**: ノード間で受け渡すフィールドは **必ず WorkflowState に宣言する**。

## 「LLMに何でもさせない」の根拠

実プロジェクトで観測した：

- **コスト**: フルエージェント化すると同じ機能で 5〜10倍のトークン消費
- **再現性**: LLM の判断は揺れる。同じ入力で違う出力が出る
- **デバッグ困難**: どこで判断が変わったか追えない
- **遅延**: ツール呼び出しの往復で5〜30秒遅くなる

→ **コードで書ける処理はコードで書く**。LLMはコードで書きにくい部分にだけ使う。

## プロンプト改善の罠 — 「追加」より「整理・削減」

実プロジェクト (cs_triage_agent v2 試行) で実証した重要な教訓:

**やりがちな失敗パターン**:
- ペルソナの人物像を立体化して長くする
- ルールを「最重要」「絶対」を多用して構造化
- 新しい概念 (例: 「顧客指定値 vs DB値」) をルールに追加
- 6セクション構成などの構造化を増やす

**結果**: 同じ評価データで LLM Judge スコア 3.52 → 3.34 (-0.18)、特に数値正確性 4.04 → 3.70 (-0.34) と劣化。プロンプト肥大化により LLM が要点を見失った。

### 取るべき改善アプローチ
| ❌ ダメ | ✅ 正解 |
|---|---|
| ルール追加で品質向上を狙う | 既存ルールの優先順位を明確化 |
| ペルソナを長く詳しく書く | ペルソナは 3-5行で人格の核だけ |
| 「最重要」「絶対」を多用 | 1-2 個だけが本当に最重要 |
| 構造化 (見出し / 番号付き) を増やす | フラットな箇条書きで 7-10 項目以内 |
| 合格件数だけ見て成功と判断 | LLM Judge で次元別スコアを比較 |

→ 詳細事例: `workspace/cs_triage_agent/v2/reports/v1_vs_v2_comparison.md`

参考: Anthropic 公式ガイドライン "Be specific, not extensive"。

## LLM as a Judge の真価

評価インフラの一部として **LLM as a Judge を早期に組み込む**ことの効用:

1. **品質の客観評価**: 単純な pass/fail を超えた次元別採点 (persona/tone/info/numerical/ng_avoidance)
2. **破滅的バグ検出**: 我々のプロジェクトで `judge.py` の本文切り詰めバグ・`make_llm` 引数バグを発見
3. **回帰検出**: プロンプト変更前後でスコア比較できるので、肥大化の罠を回避できる

→ **プロンプト変更時は必ず LLM Judge で次元別スコアを比較する**。

参考: `reference/llm_judge_skeleton.py` をプロジェクトにコピーして使う。

## 設計の核心（一言で）

> エージェントの仕事は「LLMに自由を与えること」ではなく、「LLMが活きる場所を見つけて、それ以外は決定論コードで縛ること」。

業務担当者の頭の中にある判断スキルを言語化して LLM のプロンプトに注入し、それ以外の収集・整形・配信はコードで自動化する。これがハイブリッド構成の要諦。

そして「**LLM のプロンプトに注入する**」段階で過剰に書きすぎないこと。短く・明確に・優先順位を立てる。
